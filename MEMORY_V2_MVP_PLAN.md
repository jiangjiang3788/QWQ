# 章鱼机记忆系统 V2：MVP 计划、需求解释与验收

## 1. 我对需求的理解

你需要的不是“多建几张表”，而是一套适合仿微信、多气泡回复场景的 **可控记忆引擎**：

1. **轮次符合真实体验**：用户可能连续发送一批消息，角色一次生成又拆成 10～60 个气泡；整批交互应只算一轮，同时保留消息量阈值，防止超长一轮迟迟不更新。
2. **短期记忆紧跟聊天**：当前状态、3～7 天经历、想法、重要事件、待办和生活观察可以随对话增量更新。
3. **中长期是整理，而不是实时猜测**：中期总结、成长经验、长期候选和稳定特征由总结 API 处理，并允许用户自定义每张表从聊天的哪个位置开始总结。
4. **存储与发送分离**：表格可以存很多，但每次聊天只发送固定、有效或相关的条目，避免上下文膨胀和模型混乱。
5. **用户可看见、可控制、可恢复**：普通模式突出重要字段；桌面 JSON 模式显示完整字段、策略、ID 和原始数据；移动端不暴露复杂 JSON；每张表的游标、频率和注入策略可见。
6. **先实用，再智能**：第一版先用关键词、时效、状态、重要度完成可用检索；后续再接 embedding、冲突检测和长期审核工作流。
7. **代码可维护**：轮次/策略/检索从旧的超大文件中拆为独立模块；旧数据、旧按钮和旧 UI 行为尽量兼容，不要求一次重写全项目。

## 2. 核心产品模型

### 2.1 一轮的定义

一轮不是一条消息，也不是一个气泡：

```text
上一轮角色消息结束
  → 用户连续发送的一批消息
  → 一次 AI 调用
  → AI 输出被拆成若干角色气泡
  = 1 个 memoryRound
```

自动更新支持：

- 按轮数；
- 按消息数；
- 两者先到者触发。

建议默认：**2 轮或 140 条消息先到者触发**。单次最多读取 180 条，并保留 8 条重叠。

### 2.2 两条独立链路

| 链路 | 输入 | 输出 | 原则 |
|---|---|---|---|
| 记忆更新 | 游标后的聊天、相关旧行、表格规则 | 表格增删改 | 只更新到期且相关的表 |
| 聊天注入 | 当前最近聊天、表格数据 | Prompt 记忆块 | 只注入固定、有效或相关条目 |

### 2.3 API 路由

| 记忆层 | 默认 API | 原因 |
|---|---|---|
| 短期状态/事件/待办/日常 | 主聊天 API | 高频、小范围、快速增量 |
| 中期总结/成长 | 总结 API | 需要跨记录归纳 |
| 长期候选/稳定特征 | 总结 API + 人工审核 | 避免一次对话直接形成长期人格结论 |

总结 API 未配置时，代码会回退到主 API；界面可在表级修改。

## 3. 新模板结构

| 表格 | 层级 | 更新 | 聊天注入 | 用途 |
|---|---|---|---|---|
| 核心确认档案 | 核心 | 仅手动 | 始终，重要字段 | 用户确认的核心关系、边界和稳定事实 |
| 当前状态（3～7 天） | 短期 | 2 轮或 140 条 | 有效期内 | 当前身心、需求、压力、回应策略 |
| 待办、承诺与未完成事项 | 短期 | 1 轮或 80 条 | 活跃项 Top-K | 明确事项、承诺、等待、结果 |
| 近期经历、想法与重要事件 | 短期 | 2 轮或 140 条 | 相关 Top-5 | 3～7 天内重要经历、想法、决定、困惑 |
| 日常观察 | 短期 | 2 轮或 120 条 | 相关 Top-4 | 睡眠、饮水、身体、精力、情绪 |
| 中期总结与成长经验 | 中期 | 手动/独立游标 | 相关 Top-3 | 周期总结、成长变化、可复用经验 |
| 长期候选审核队列 | 审核 | 手动 | 不注入 | 长期结论的缓冲区 |
| 稳定长期特征库 | 长期 | 手动批准 | 相关 Top-6 | 偏好、规律、有效安抚、价值观、世界观 |

## 4. 版本计划

### V2.0 MVP：先可用

**目标：** 消除全表发送和单一游标，建立完整最小闭环。

已实现：

- `memoryRound` 轮次记录；一批 AI 气泡只记一轮；
- 轮数/消息数/先到者三种触发；
- 每张表独立游标、独立更新策略；
- 可把游标设到消息序号 0～最新；
- 顶部更新按钮只更新当前选中表；
- 自动更新只处理到期表，每次默认最多两张；
- 单次更新按最大消息数分块，并支持重叠；
- 更新模型只看目标表与最多约 12 条候选旧行；
- 聊天时按 `always / active / relevant / never` 注入；
- 全局和表级字符预算；
- 普通模式只显示重要字段；
- 桌面端 JSON 模式显示完整字段、ID、策略和原始 JSON；
- 移动端强制普通模式；
- 模板可视化编辑器同步支持层级、频率、API、注入和字段重要性；
- 完整迁移原 32 个键值与 206 条原始行。

**验收：**

1. AI 一次返回 30 个气泡，轮数只增加 1。
2. 当前状态到 2 轮或 140 条消息时可自动更新。
3. 中期表可把游标设到任意消息序号后手动总结。
4. 日常聊天 Prompt 不再出现全部 179 条长期记录，只出现相关 Top-K。
5. 手机宽度下 JSON 按钮不可用，完整结构不显示。
6. 原模板 206 条行记录全部可在新模板 JSON 模式中追溯到“原始记录ID”。

### V2.1：总结审核体验

**目标：** 让中长期总结可预览、可确认，而不是直接落库。

计划：

- 更新前显示范围预览：起点、终点、轮数、消息数；
- 总结 API 输出先进入差异预览；
- 支持逐条接受、拒绝和编辑；
- 一键把中期总结生成长期候选；
- 一键批准候选进入稳定长期特征；
- 长期删除和覆盖必须二次确认；
- 提供“为什么召回这条记忆”的可解释视图。

### V2.2：检索质量

**目标：** 从关键词 MVP 升级为混合检索。

计划：

- 表格行 embedding 索引；
- 语义、关键词、时间、重要度、状态的混合评分；
- 相似行自动查重与合并建议；
- 每表配额和跨表重排；
- Prompt 实际注入内容与字符量诊断；
- 索引重建和模型切换兼容。

### V2.3：长期可靠性

**目标：** 防止长期记忆漂移和相互矛盾。

计划：

- 来源消息 ID 与证据跨度；
- 反例、冲突和替代关系；
- 长期特征版本历史；
- 自动过期与归档策略；
- 数据迁移版本校验；
- 回滚到任意记忆快照。

## 5. UI 流程

### 普通模式

面向日常使用：

- 左侧/横向表格导航；
- 只显示 `important: true` 的字段；
- 表格像电子表格一样直接编辑；
- 只暴露常用操作：更新当前表、新增行、删除行、调整游标。

### JSON 模式（仅桌面）

面向模板作者和排错：

- 显示所有字段；
- 显示模板/表/字段/行 ID；
- 显示更新策略和注入策略；
- 显示整张表的 schema、data、lockedFields；
- 不在手机端开放，避免误触和窄屏不可用。

## 7. 代码结构

| 文件 | 职责 |
|---|---|
| `js/modules/memory_table_policy.js` | 轮次、策略、独立游标、相关性评分、预算 |

## 8. 当前 MVP 的明确边界

- 相关性检索目前是中文 n-gram/关键词 + 时效 + 重要度 + pinned，不是 embedding。
- 每日/每周定时不是后台服务；应用关闭时不会运行，下次打开或聊天时补做。
- 历史聊天在升级前没有 `memoryRoundId`，因此初始轮数为 0；消息数兜底仍可工作。
- 总结尚无逐条差异审核界面；V2.0 会直接应用合法 XML 更新。
- 已完成静态语法和项目静态检查；未在真实 API、所有手机浏览器上做端到端回归。
