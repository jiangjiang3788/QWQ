# 章鱼机 V2.14-R2：正式写入门禁与统一回执说明

## 一、版本定位

V2.14-R2 基于 V2.14-R1，将“统一写入协调器”从可选的保存工具收敛成正式记忆的架构边界。

本版仍按单人自用设计，不引入多人锁、远程 revision、三方合并或分布式事务。目标是用较低复杂度确保：

- 正式记忆的数据变化由 `Domain` 执行；
- 高风险业务用例由 `WriteGateway` 提交；
- 同一角色的持久化仍由 `WriteCoordinator` 串行与回滚；
- 所有成功写入返回统一、可理解的操作回执；
- 后续代码若绕过正式入口，静态门禁直接失败。

## 二、三层正式写入结构

```text
业务用例
表格编辑 / Sidecar / 审核 / 长期晋升
        ↓
WriteGateway
统一提交、统一回执、操作详情记录
        ↓
WriteCoordinator
同角色串行、快照、持久化、失败回滚
        ↓
Domain
正式表格数据新增、修改、删除、替换
```

三层职责不同：

- `Domain` 只负责正式数据结构的合法变更；
- `WriteCoordinator` 只负责本地单人场景的写入顺序和保存失败恢复；
- `WriteGateway` 负责把一次业务变更转成一致的操作结果。

本版没有建立第二套数据层。

## 三、统一操作回执

新增：

```text
js/features/memory/write_gateway.js
```

每次正式提交统一返回：

```text
schemaVersion
producerVersion
transactionId
operationId
characterId
source
action
status
recordCount
fieldCount
templateId / tableId / rowId
summary
changes
persisted
rollbackApplied
```

计数规则与用户界面保持一致：

```text
同一行修改 14 个字段
→ 1 条记忆
→ 14 个字段变化
```

回执可写入现有操作详情中心，但不会为了记录回执额外修改正式记忆。

## 四、Domain 正式写入能力补齐

本版增加：

- `replaceFormalData()`：恢复或替换完整正式记忆数据；
- `replaceTemplateData()`：安全替换单个模板的数据；
- `setRowTagBundle()`：通过 Domain 修改整行标签。

审核回滚、历史恢复、导入覆盖和标签编辑不再各自直接覆盖正式数据。

## 五、已接入写入网关的主链

- 表格单字段与标签编辑；
- 全屏整行编辑和撤销；
- 行新增、删除和移动；
- 聊天 Sidecar 正式更新；
- Sidecar 候选保存、合并、忽略和删除；
- AI 直接更新正式表；
- 更新审核应用、拒绝、取消和恢复；
- 长期候选批准晋升；
- 日记转换档案；
- 模板数据安全导入。

Sidecar 内部仍允许在事务快照中组装变更，但正式提交必须经过写入网关；它不能独立持久化半完成结果。

## 六、架构门禁

`tools/check_memory_architecture.py` 新增正式写入检查：

- 主控制器不能直接替换 `chat.memoryTables.data`；
- 网关客户端不能直接调用 `WriteCoordinator.run()`；
- 正式字段不能在业务模块中直接写入 `row.cells[fieldId]`；
- 网关客户端必须显式依赖 `writeGateway`；
- Sidecar 必须由网关保护后才能提交。

少数允许直接接触数据结构的底层模块被写入架构合同白名单，包括 Domain、导入适配器、协调器和 Sidecar 变更生成器。白名单不代表可以自行持久化。

## 八、单人版边界

本版刻意不实现：

- 多人协作锁；
- 多设备远程 revision；
- 服务端 CAS；
- 三方冲突合并；
- 分布式事务日志。

当前保障范围是同一运行实例、同一角色内的串行提交和失败回滚。下一版可进入字段级写入策略，解决“整张表允许直接写，但部分字段其实属于推断或角色运行态”的可信度问题。
