# 章鱼机 V2.11-R7：表格缓存与合并持久化说明

## 一、本版目标

V2.11-R7 不增加新的记忆页面、卡片或侧栏，也不改变记忆模板和备份格式。

本版只处理 V2.11-R6 后仍然存在的四个工程问题：

1. 表格筛选、搜索和虚拟窗口刷新时会重复组装相同的派生模型；
2. 连续编辑多个字段会重复触发完整角色持久化；
3. 字段修改缺少轻量、可预期的撤销能力；
4. 新增、删除、移动和字段提交仍有部分操作细节停留在主控制器。

目标是减少重复计算与重复写入，同时保持 R0—R6 已建立的扁平界面、虚拟表格和局部刷新能力。

## 二、表级派生模型缓存

新增：

```text
table_cache.js
```

缓存范围按以下三项隔离：

```text
角色/聊天 ID + 模板 ID + 表 ID
```

缓存只保存表格渲染所需的派生模型，不复制完整角色对象和聊天历史。

缓存签名包含：

- 表级修订号；
- 行顺序和行 ID；
- 更新时间；
- 生命周期状态；
- 固定和标签锁定状态；
- 主题、场景和主体标签；
- 最近一条记忆历史标识。

字段、标签、生命周期或结构发生变化后，会对当前表执行显式失效。整页重新进入记忆工作区时，会失效当前角色的表格缓存，避免其他记忆模块直接修改数据后出现陈旧视图。

缓存采用最多 32 项的 LRU 策略，不会无限增长。

## 三、连续保存合并

新增：

```text
table_persistence.js
```

字段编辑按照角色 ID 进入串行保存队列。短时间内同一角色的多个字段提交，会合并为一次底层保存调用。

例如真实数据核验中的三次连续字段提交：

```text
字段提交 1 ┐
字段提交 2 ├─ 合并为 1 次实际写入
字段提交 3 ┘
```

结构变更仍然立即保存，包括：

- 新增行；
- 删除行；
- 移动行；
- 撤销字段编辑。

保存队列还提供：

- 同一角色严格串行写入；
- 保存失败向调用方传播；
- 页面进入后台时主动刷新待保存内容；
- 存在待保存内容时阻止直接关闭页面；
- 保存请求、实际写入、合并数量和失败次数指标。

本版没有修改底层 IndexedDB schema，也没有创建另一份记忆存储。

## 四、最近字段编辑撤销

新增：

```text
table_editor.js
```

字段提交会记录紧凑撤销项：

- 模板 ID；
- 表 ID；
- 行 ID；
- 字段 ID；
- 修改前值；
- 修改后值；
- 操作时间。

每个角色最多保留最近 20 次字段编辑。撤销后立即持久化，并使当前表缓存失效。

撤销记录只存在于当前运行会话，不写入角色数据和 `.ee` 备份，因此不会增加备份体积。

当前撤销范围仅包含字段值修改，不包含删除整行、合并记忆和生命周期批量操作。这样可以避免把高风险结构操作伪装成轻量撤销。

## 五、编辑控制器边界

新增：

```text
table_edit_controller.js
```

该模块统一处理：

- 字段输入提交；
- 新增行；
- 删除行；
- 上下移动；
- 撤销编辑。

`memory_table.js` 不再直接调用行编辑服务的新增、删除和移动实现，只保留页面级事件入口和工作区编排。

## 七、真实数据浏览器核验

核验使用用户提供的清理版 `.ee` 的安全副本：

- 角色：阿沉；
- 聊天消息：19,133 条；
- 记忆表：8 张；
- 核验表：稳定长期特征库；
- 数据行：179 条。

浏览器执行流程：

1. 使用真实八表模板和 179 条长期记忆渲染；
2. 滚动到虚拟表格中段；
3. 进入一条可见记录编辑；
4. 连续提交三个字段；
5. 完成编辑并刷新当前表格区域；
6. 执行一次撤销；
7. 再次命中相同派生模型缓存；
8. 核对顶栏、左侧表清单、表格挂载点和滚动位置。

结果：

| 指标 | 桌面 | 390px 手机 |
|---|---:|---:|
| 数据总行数 | 179 | 179 |
| 当前 DOM 数据行 | 17 | 17 |
| 保存请求 | 4 | 4 |
| 实际物理写入 | 2 | 2 |
| 合并写入数量 | 2 | 2 |
| 缓存命中 | 3 | 3 |
| 缓存未命中 | 4 | 4 |
| 虚拟滚动位置保持 | 是 | 是 |
| 页面顶栏节点保持 | 是 | 是 |
| 左侧表清单节点保持 | 是 | 是 |
| 表格挂载节点保持 | 是 | 是 |
| 页面级横向溢出 | 无 | 无 |
| 浏览器运行错误 | 0 | 0 |

四次保存请求的组成是：三次连续字段提交合并为一次实际写入，随后一次撤销产生一次立即写入，因此总计两次实际写入。

桌面文档宽度与视口均为 1440px；手机文档宽度与视口均为 390px。手机宽表只在表格内部横向滚动，不会撑宽整个页面。

所有测试修改均作用于浏览器中的数据副本，没有写回原始 `.ee` 文件。

## 十、下一阶段建议

下一版仍不应增加新功能框。优先方向应是：

- 将模板设计器、导入导出和历史视图继续移出 `memory_table.js`；
- 建立统一的表格 mutation 事件，使候选晋升、批量治理和外部关系操作也能精确失效缓存；
- 为删除和合并等高风险操作建立独立可恢复快照，而不是扩大轻量字段撤销的职责；
- 将主控制器门禁继续降低到 4,200 行以内。
