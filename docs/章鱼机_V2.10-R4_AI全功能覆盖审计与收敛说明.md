# 章鱼机 V2.10-R4：AI 全功能覆盖审计与收敛说明

## 1. 本版目标

V2.10-R4 以 V2.10-R3.3 为稳定基线，不重写聊天、记忆、小剧场或图片系统，而是继续完成“所有 AI 行为可解释、可追踪”的主线。

本版重点解决：

1. 低频 AI 功能虽然经过统一请求运行时，但在操作中心仍显示成笼统的“执行 AI 功能”；
2. 部分请求只有 HTTP 成功记录，没有业务结果是否真正保存的回执；
3. 新增 AI 请求可能忘记登记，长期重新出现不可追踪调用；
4. 用户缺少一个能查看当前 AI 能力覆盖情况的入口。

## 2. 本版完成内容

### 2.1 新增统一 AI 能力目录

新增：

```text
js/modules/ai_capability_catalog.js
```

目录目前登记 27 种产品操作，分为聊天、小剧场、日记、记忆、图片识别、图片生成、通话、互动、关系和兜底等类别。

每个能力只保存产品元数据：

- 操作类型；
- 用户可见标题；
- 分类；
- 图标；
- 已知请求任务名；
- 必要的任务匹配规则。

它不保存 Prompt、角色、聊天或记忆对象，也不承担业务实现。

### 2.2 请求运行时自动映射为产品操作

`OVOAIRequestRuntime` 在创建隐式操作前，会先经过能力目录解析。

例如：

| 请求任务 | 操作中心显示 |
|---|---|
| `avatar-recognition` | 识别头像内容 |
| `sticker-recognition` | 识别表情包内容 |
| `image-description` | 识别聊天图片 |
| `gpt-image-generation` | 生成图片 |
| `novelai-image-generation` | 使用 NovelAI 生成图片 |
| `legacy-video-call` | 生成通话回复 |
| `legacy-call-summary` | 生成通话总结 |
| `battery-interaction` | 生成电量互动 |
| `block-system` | 判断拉黑与好友申请 |
| `vector-embedding` | 生成向量记忆索引 |
| `memory-table-embedding` | 生成结构化档案向量 |
| `journal-generation` | 生成回忆日记 |

因此低频能力即使暂时没有独立页面，也不会再全部混入一个没有语义的通用操作。

诊断记录增加 `capabilityType`，同时提供 `getCapabilityCoverage()`，可统计最近真实调用的能力和请求数量。

### 2.3 操作中心新增“功能覆盖”页面

全屏悬浮球新增“功能覆盖”入口，展示：

- 已登记的产品操作数量；
- 最近实际使用的能力数量；
- 最近网络请求数量；
- 仍落入通用兜底的请求数量；
- 按分类列出的能力；
- 每种能力最近是否调用、调用次数和最近状态。

这个页面是审计视图，不会修改业务数据。

### 2.4 自动图片描述成为聊天子操作

聊天中的后台自动识图现在会：

1. 作为当前聊天回复的子操作启动；
2. 记录待识别图片数量；
3. 将真实识图请求绑定到该子操作；
4. 等待模型返回并解析描述；
5. 将描述写回原聊天消息；
6. 保存原聊天；
7. 记录 `chat_message` 数据变化回执；
8. 在数据真正保存后才标记完成。

因此“请求成功”和“图片描述已写回聊天”不再混为一谈。

### 2.5 表情包批量识别增加完整业务回执

批量识别现在是独立的 `vision.sticker.batch` 操作：

- 显示总数量和当前进度；
- 每个识别请求绑定到同一批次；
- 每条识别结果写回表情包后记录 `sticker` 更新回执；
- 最终显示成功、失败和取消数量；
- 用户取消后不会把未处理项误报为成功。

### 2.6 手动日记生成增加数据变化回执

手动生成日记现在显式创建 `journal.generate` 操作，记录：

- 日记生成要求；
- 使用的聊天范围；
- 真实请求；
- 新日记写入；
- 新增日记 ID、标题和消息范围；
- 失败原因。

自动日记继续作为聊天的后台子操作，并把父操作 ID 传入日记请求，避免产生游离的顶层通用记录。

### 2.7 电量互动改为完整操作生命周期

电量互动不再只记录一次请求，而是记录完整业务流程：

- 读取电量条件；
- 选择目标角色；
- 生成互动内容；
- 解析模型输出；
- 展示互动结果；
- API 不可用时明确标记跳过；
- 解析或请求失败时明确标记失败。

### 2.8 拉黑判断与好友申请增加结果回执

拉黑系统的两条 AI 链路现在会：

- 显式创建 `safety.block.check` 操作；
- 绑定真实判断或申请理由请求；
- 在生成好友申请后记录 `friend_request` 新增回执；
- 记录下一次检查时间的设置变化；
- 在实际保存后完成操作；
- 请求或解析失败时记录失败。

### 2.9 其他低频能力补充产品元数据和 Prompt 来源

本版还补充了以下请求的专属类型、阶段或来源：

- 头像识别；
- 单个表情包识别；
- GPT 图片生成；
- NovelAI 图片生成；
- 视频通话回复；
- 通话总结；
- 向量记忆摘要与 Embedding；
- 结构化档案 Embedding；
- 日记合并整理。

## 3. 覆盖门禁

新增测试：

```text
tests/run_v210_r4_ai_capability_coverage_checks.js
```

测试会扫描生产 JavaScript 中所有字面量 `task:` 和 `runtimeTask:`：

- 若某个任务没有登记，且只能解析为 `ai.request`，测试直接失败；
- 校验能力目录、操作运行时和请求运行时版本；
- 校验能力目录在运行时之前加载；
- 校验隐式头像识别会生成正确的产品操作；
- 校验覆盖统计 API；
- 校验自动识图、批量表情识别、日记、电量和好友申请回执代码仍存在。

当前源码共扫描到 17 个字面量 AI 任务，全部有明确映射。

这不是完全替代人工审计，但能防止最常见的“新增请求后忘记登记”。动态拼接任务仍需要代码评审和运行时兜底统计共同发现。

## 5. 数据与隐私边界

本版继续遵守原操作中心边界：

- 不复制完整角色对象；
- 不复制完整聊天数组；
- 不复制记忆任务队列或 runtime；
- API Key、Authorization、Token 等仍由请求运行时脱敏；
- Prompt 来源只保存受控文本和摘要；
- 数据变化回执只保存必要前后值、字段和实体引用；
- 操作历史仍受数量和正文长度限制。

## 8. 明确限制

1. 本环境没有使用你的真实外部模型 API，因此不同 Provider 的真实响应格式仍需部署后分别试用。
2. 浏览器自动 UI 烟雾测试尝试启动，但运行环境阻止访问本地页面，返回 `ERR_BLOCKED_BY_ADMINISTRATOR`；因此本版不宣称完成了 Chromium 页面渲染验证。
3. 当前静态门禁主要覆盖字面量任务名；动态生成的任务名依靠运行时“通用兜底数量”发现。
4. 网页真正刷新或关闭后，尚未返回的网络请求仍受浏览器生命周期限制，不能凭前端代码恢复；持久化记忆任务仍按 R3.3 逻辑恢复。
5. 大文件拆分和正式数据库 schemaVersion 迁移不属于本版，建议在 V2.11 单独处理，避免 R4 同时改动数据格式。

## 9. 部署后建议复测

至少各执行一次：

1. 带图片的私聊回复，确认“识别聊天图片”显示为聊天子操作，并出现消息修改回执；
2. 批量识别 2—3 个表情包，确认每条保存回执和批次统计；
3. 手动生成一篇日记，确认新增日记回执；
4. 触发一次头像识别和图片生成，确认不再显示为笼统操作；
5. 打开操作中心“功能覆盖”，确认最近调用次数和通用兜底数量；
6. 使用实际 Provider 核对最终原始请求和来源视图。
