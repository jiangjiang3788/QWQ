# 章鱼机 V2.11-R4：统一待处理与长期记忆治理说明

## 一、本版目标

V2.11-R4 不继续增加新的常驻卡片或侧栏，而是把 R0—R3.1 已经存在的审核、候选、生命周期和任务入口收敛成一个统一工作流。

本版解决三个问题：

1. “待处理”原先由多张入口卡片组成，用户仍需要在审核、候选、任务、反馈等页面之间寻找真正需要处理的事项；
2. 179 行长期记忆缺少轻量筛选，查看待复核、固定、归档或某类标签时仍需手工搜索；
3. 长期候选晋升与表格渲染继续留在四千多行的主控制器中，新增功能会再次推动单文件膨胀。

## 二、统一待处理队列

### 2.1 单一列表

“待处理”首页不再使用入口卡片网格，改为一条按风险和时间排序的扁平列表。统一扫描：

- 结构化记忆更新批次；
- 长期候选审核队列；
- `uncertain`、`conflicting`、`expired` 或达到复核日期的记忆；
- 失败、排队和暂停的后台任务；
- 短期记忆候选；
- 记忆召回反馈。

列表只保留一层行结构：选择、摘要、操作。没有在列表项内继续套卡片。

### 2.2 风险排序

优先级规则：

- 高风险：冲突、过期、失败任务、含高风险建议的审核批次；
- 中风险：普通更新批次、长期候选、可信度待确认或到期复核；
- 低风险：排队任务、短期候选和召回反馈等系统入口。

同优先级按更新时间倒序排列。

### 2.3 统一操作

队列支持：

- 进入结构化更新审核；
- 批准或拒绝长期候选；
- 打开记忆详情；
- 确认记忆继续有效；
- 延后 30 天复核；
- 归档失效记忆；
- 对可安全批量处理的复核项执行批量确认、延后或归档。

存在未解决冲突的记录不能被批量确认，必须进入详情单独处理。

## 三、长期记忆轻量筛选

行式表格顶部新增一条单层筛选栏：

- 全部；
- 待复核；
- 有效；
- 固定；
- 归档/替代；
- 标签搜索。

筛选直接复用现有生命周期和标签结构：

```text
row.meta.lifecycle
row.meta.status
row.meta.pinned
row.meta.tagBundle
row.meta.tags
```

没有创建第二套筛选状态或标签字段。

“待复核”同时覆盖：

- `uncertain`；
- `conflicting`；
- `expired`；
- 已达到 `reviewAt`；
- 已达到 `expiresAt`。

## 四、长期候选晋升

长期候选处理从主控制器拆到 `candidate_service.js`。

批准候选时：

1. 查找同模板中的长期行表；
2. 通过原始记录 ID 和正文检查重复；
3. 只写入目标表已有字段；
4. 保留候选类别、置信度、支持证据和例外；
5. 将来源标记为长期候选审核或成长沉淀；
6. 将新长期记忆标记为用户确认和有效；
7. 更新候选审核状态；
8. 写入现有历史记录并清理检索缓存。

没有新增候选数据库，也没有改变模板或备份结构。

## 五、代码收敛

新增五个职责单一的模块：

```text
candidate_service.js       156 行
 table_filter.js            80 行
 governance_queue.js       216 行
 governance_controller.js  129 行
 table_grid.js               75 行
```

职责分别为：

- 候选状态与长期晋升；
- 生命周期和标签筛选；
- 待处理事项扫描、排序与纯渲染；
- 队列动作与批量处理；
- 键值表、行表和原始 JSON 的纯渲染。

从 `memory_table.js` 移出的主要生产路径：

- 长期候选批准与状态写入；
- 键值表渲染；
- 行式表格渲染；
- 原始 JSON 渲染；
- 待处理队列的扫描与操作逻辑。

主控制器从 R3.1 的 4,518 行降至 4,416 行，减少 102 行；新增能力没有重新塞回主文件。

## 七、界面核验

### 7.1 统一队列

桌面 1440×1000 和手机 390×844：

- 页面宽度与视口宽度一致；
- 页面级横向溢出为 0；
- 实际渲染 4 个队列项；
- 仅有 1 个搜索输入；
- 没有嵌套入口卡片。

### 7.2 长期记忆表格

桌面：

- 179 行一次加载；
- 5 个生命周期筛选；
- 表格宽度适配可用区域；
- 普通阅读模式单元格编辑器为 0。

手机：

- 页面本身没有横向溢出；
- 宽表在表格自己的滚动区域内横向浏览；
- 页面宽度保持 390px；
- 普通阅读模式单元格编辑器仍为 0。

界面中的 179 个选择控件是每行现有的紧凑操作菜单，不是单元格编辑器。下一阶段仍可通过行虚拟化或按需创建操作菜单继续降低 179 行表的 DOM 数量。

## 十、已知边界与下一步

R4 已经把入口和控制器继续收敛，但 179 行表仍一次创建 179 个行操作菜单。下一版更适合做：

- 行虚拟化或分段加载；
- 行操作菜单按点击时创建；
- 治理队列的状态持久化和完成记录；
- `memory_table.js` 中剩余事件分发和页面编排继续拆分。

下一版不应再增加新的常驻卡片，而应优先减少大表 DOM 和主控制器职责。
