# 章鱼机 V2.10-R1：AI 操作来源视图实施说明

## 1. 版本目标

V2.10-R1 延续 R0 的 AI 操作中心，不新增一套平行的聊天、记忆或小剧场系统。本版解决的问题是：

> 用户能够看到最终模型请求，也应当知道请求中的内容分别来自角色档案、用户档案、世界书、记忆、聊天历史、本次输入还是输出规则。

R1 同时保留两层证据：

1. **来源视图**：按照产品概念拆分，便于普通用户阅读；
2. **最终原始请求**：展示发送前的真实 payload，用于高级核对。

来源视图不会替代最终请求，也不会根据设置页面重新猜测一份 Prompt。

---

## 2. 用户可见变化

打开悬浮球中的某次操作后，每个模型请求现在包含“发送内容来源”：

- 系统规则；
- 角色档案；
- 用户档案；
- 世界书；
- 结构化记忆；
- 日记记忆；
- 向量记忆；
- 聊天历史；
- 本次输入；
- 任务要求；
- 输出规则；
- 工具与模型参数；
- 其他上下文。

每个来源可以展开查看：

- 来源正文；
- 字符量；
- 是否实际发送；
- 来源或选择理由；
- 世界书等复合来源中的子条目；
- 未发送条目的排除原因。

状态含义：

| 状态 | 含义 |
|---|---|
| 实际发送 | 直接来自最终请求，或业务模块明确记录了实际拼入请求的文本 |
| 已核对 | 来源文本已经与最终 Prompt 做过存在性核对 |
| 参与组装 | 业务模块报告它参与了组装，但无法逐字符证明最终形态 |
| 未发送 | 候选存在，但本次没有进入最终请求 |

悬浮球仍提供“查看最终原始请求”，用户可以核对 Provider 最终接收到的 messages、contents、system_instruction 和生成参数。

## 3. 新增统一 Prompt 来源层

新增文件：

```text
js/modules/prompt_trace.js
```

全局入口：

```js
OVOPromptTrace.build(finalRequestBody, explicitSources, metadata)
```

它只负责只读追踪，不参与 Prompt 决策，不修改聊天、记忆或世界书数据。

### 3.1 输入

- `finalRequestBody`：发送给模型前的最终请求体；
- `explicitSources`：业务模块在组装过程中报告的来源；
- `metadata`：任务、调用来源、Provider 和模型信息。

### 3.2 输出

```js
{
  version: 'prompt-trace.v1',
  capturedAt,
  task,
  source,
  provider,
  model,
  sections: [],
  summary: {
    sectionCount,
    sentSectionCount,
    sourceChars,
    byType: []
  }
}
```

### 3.3 合并原则

1. 业务模块的显式来源优先；
2. 最终 system 消息始终保留为完整核对层；
3. 未提供显式来源时，从最终请求自动识别聊天历史、本次输入和生成参数；
4. 不重复生成同类型的推测来源；
5. 来源视图与最终原始 payload 同时保留。

## 4. 首批深度覆盖

### 4.1 私聊角色回复

`chat_ai.js` 新增私聊来源构建器，记录：

- 当前角色名称、状态和有效人设；
- 与角色绑定的用户称呼和用户人设；
- 本次世界书诊断中命中、注入、截断或排除的条目；
- 当前启用的长期记忆模式：结构化记忆、向量记忆或日记记忆；
- 从最终 system Prompt 提取的回复格式和功能约束；
- 从最终请求自动识别的完整系统提示词；
- 实际聊天历史；
- 最后一条本次输入；
- 温度、流式设置、工具等生成参数。

世界书子条目会保留：

- 名称；
- 是否注入；
- 注入字符量；
- 是否截断；
- 命中关键词；
- 注入位置；
- 排除原因。

### 4.2 手动小剧场

文字与 HTML 小剧场记录：

- 对应模式的输出规则；
- 用户输入的剧情要求；
- 本次选中的角色档案；
- 用户人设预设；
- 用户主动选择的世界书；
- 选中的聊天上下文；
- 角色日记记忆；
- 最终发送的 user Prompt。

### 4.3 角色主动创作小剧场

记录：

- 角色档案；
- 用户人设；
- 角色绑定世界书；
- 最近日记总结；
- 设置范围内的最近聊天；
- 自定义创作要求；
- HTML 或文字输出规则。

### 4.4 结构化档案更新

现有结构化记忆更新使用一条组合 Prompt。R1 从最终 Prompt 中识别：

- 提取和 XML 输出约束；
- 角色信息；
- 用户信息；
- 目标模板、字段规则和当前数据；
- 本次实际读取的聊天范围。

这项解析只用于展示，不改变原有 MemoryTable/OvoMemory 的提取、审核和写入逻辑。

### 4.5 其他 AI 功能

没有专属来源适配器的功能仍能得到通用来源视图：

- 完整系统消息；
- 历史 user/assistant 消息；
- 最后一条非控制用户输入；
- 后台继续对话等控制指令；
- 生成参数和工具配置。

后续版本可逐项增加业务来源适配器，不需要修改悬浮球 UI 协议。

## 5. 与现有架构的关系

R1 没有推倒 R0 或现有业务模块：

```text
业务功能
  ├─ chat_ai.js
  ├─ theater.js
  └─ memory_table.js / OvoMemory
          │
          ▼
OVOAIRequestRuntime
          │ 最终请求体 + promptSources
          ▼
OVOOperationRuntime.attachRequest()
          │
          ▼
OVOPromptTrace.build()
          │
          ├─ 来源视图
          └─ 最终原始请求
          │
          ▼
QuickDock / Proment
```

职责边界：

- `OVOAIRequestRuntime`：请求传输、取消、超时和诊断；
- `OVOOperationRuntime`：一次完整功能操作的生命周期；
- `OVOPromptTrace`：将最终请求和业务来源转换成统一只读结构；
- `QuickDock`：面向用户展示；
- `Proment`：Prompt 与记忆的深度工作台；
- `OvoMemory`：仍是记忆系统唯一对外门面。

## 6. Proment 收敛

Proment 的“最近真实 Prompt”现在优先读取最近一次操作中的 `promptTrace`，并按来源展示。

旧的 `__ovoLastPromptSnapshot` 仍作为兼容回退，避免已有功能突然失效。后续所有主要功能完成迁移后，再删除旧快照入口。

这意味着：

- 悬浮球负责本次操作的快速解释；
- Proment 负责更深的 Prompt 检查和设计对照；
- 两者读取同一份操作来源记录，不再分别推测。

---

## 7. 数据控制与安全

R1 延续 R0 的 sessionStorage 策略：

- 最多保留 100 次操作；
- 最近 12 次保留详细请求和来源正文；
- 更早记录只保留来源结构和摘要；
- 单段来源正文有长度上限；
- 来源子条目有数量上限；
- 原始请求预览有长度上限；
- API Key、Token、Authorization、密码等字段替换为 `***`；
- 操作记录不进入角色数据库和正式备份；
- 不保存完整角色数据库、完整记忆表或任务队列副本；
- 安全克隆器仍有层级、数组和字段上限。

本版 UI 烟雾测试中发现深层来源摘要被过早折叠，已将受控克隆深度调整到可完整保留 `promptTrace.sections/items/summary`，同时继续阻断无限递归。

## 9. 兼容性

- 不改变 IndexedDB schema；
- 不改变角色、聊天、记忆或世界书的数据格式；
- R0 的 sessionStorage 操作记录可以继续加载；
- 新记录使用 `schemaVersion: 2`；
- 没有 `promptTrace` 的旧记录仍可查看原始请求；
- 不改变模型请求正文，因此不应改变生成结果；
- OpenAI-compatible 与 Gemini 两类请求结构均支持通用拆分。

---

## 10. 已知边界

1. R1 展示的是“发送内容与来源”，尚未展示操作完成后的全部数据变更；
2. 自动日记、自动向量记忆、自动小剧场等仍未形成完整父子操作树；
3. 通用拆分只能识别消息角色，无法理解每个功能内部的所有业务语义；
4. 私聊中的角色档案和记忆使用存在性核对，不等同于逐字符位置映射；
5. 来源字符量可能包含“分类来源 + 完整 system 核对层”的重复内容，因此不能直接当作 Token 估算；
6. 尚未使用用户的真实外部 API 服务进行端到端调用。

用户需要判断“模型最终收到什么”时，应查看最终原始请求；需要判断“这些内容为什么出现”时，应查看来源视图。

## 12. 下一版建议：V2.10-R2

R2 应进入“操作结果与后台子任务”阶段：

1. 给聊天回复后的自动记忆、日记、向量和小剧场建立父子操作关系；
2. 在操作详情中显示“已执行、未达到条件、跳过、失败”；
3. 增加数据变更回执：新增消息、修改档案字段、新增记忆行、保存小剧场；
4. 让用户看到一次聊天之后系统在后台又做了什么；
5. 保持 UI 与逻辑同步交付，不先做不可见的大规模重构。
