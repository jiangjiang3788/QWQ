# 章鱼机 V2.10-R3.2 档案记忆注入修复说明

## 1. 结论

用户反馈“完全没看到档案记忆放入上下文”是正确的。

对 V2.10-R3.1 最终发布包与实际备份数据重新核对后，确认旧逻辑把 `journal / table / vector` 当成互斥的三选一模式。角色选择“向量记忆”后，即使结构化档案已经启用、绑定模板并有大量有效数据，结构化档案仍会被聊天 Prompt 构建器直接排除。

R3.1 只修复了自定义 Prompt 缺少 `{{共同回忆}}` 时的兜底注入，并把 `memory_live_context` 显示为“档案记忆”。但 `memory_live_context` 实际只是实时状态与待办，不是结构化档案正文，因此上一版的判断和命名都不准确。

## 2. 实际备份核验

主角色当前配置：

- 当前补充模式：`vector`
- 结构化档案：已启用
- 已绑定结构化模板：1 个，且模板真实存在
- 最近结构化档案上下文：约 2,880 字符
- 结构化档案数据体：约 466 KB（JSON 文本口径）
- 向量记忆条目：0
- 最近向量上下文：0 字符
- 收藏日记：0

所以旧版运行结果是：选择了向量模式，但向量为空；结构化档案又因互斥逻辑被跳过，最终长期档案上下文基本为空。

## 3. 修复后的记忆语义

R3.2 将长期记忆改为分层组合：

| 角色设置 | 最终真实上下文 |
|---|---|
| 仅结构化档案 | 结构化档案 |
| 档案 + 回忆日记 | 结构化档案 + 收藏日记 |
| 档案 + 向量记忆 | 结构化档案 + 向量检索结果 |

结构化档案现在是角色的基础档案。`memoryMode` 只决定额外补充来源，不再决定是否读取结构化档案。

只有以下情况不会注入结构化档案：

- 用户明确关闭结构化档案；
- 当前角色没有绑定任何结构化模板；
- 绑定模板中没有符合注入策略的有效内容。

## 4. 代码链路修复

### 4.1 收敛到现有记忆门面

聊天模块优先通过：

```text
window.OvoMemory.context.prepare()
window.OvoMemory.context.get()
```

读取结构化档案，不再另建平行记忆系统。旧的全局兼容函数只作为回退。

### 4.2 非 table 模式允许读取基础档案

`memory_table.js` 增加受控的 `allowInactiveMode` 参数：

- 不改变原有管理页面和历史接口行为；
- 只允许聊天的分层记忆入口在 vector/journal 模式下读取结构化档案；
- `force` 仍保持“完整档案导出/整理”的原语义，不被滥用。

### 4.3 统一分层上下文构建

新增统一函数：

```text
prepareCombinedLongTermMemoryContext()
buildCombinedLongTermMemoryContext()
```

私聊、后台回复、总结、剧情节点、通话总结和自定义 Prompt 都使用同一套组合逻辑。

### 4.4 最终请求防线

在 `generatePrivateSystemPrompt()` 完成后、Provider 请求体创建前，再执行一次最终检查：

- 结构化档案启用且绑定模板；
- 最终 system prompt 中却没有结构化档案标签；
- 系统自动追加结构化档案区块。

这可以防止未来某个自定义模板、剧情节点分支或重构再次静默丢失档案。

### 4.5 明确的最终 Prompt 标签

真实请求现在使用独立标签：

```xml
<structured_archive_memory>...</structured_archive_memory>
<vector_memory_context>...</vector_memory_context>
<journal_memory_context>...</journal_memory_context>
```

悬浮球从最终 system prompt 中读取这些标签，不再根据设置推测。

### 4.6 纠正实时状态的命名

`memory_live_context` 改为显示：

> 实时档案状态与待办

它不再被误称为“结构化档案记忆”。

实时状态、待办和 Sidecar 也不再因为选择向量/日记补充模式而被关闭；只要结构化档案启用并绑定模板，就按自身开关运行。

## 5. UI 修正

所有“三选一”旧文案已替换为分层语义：

- 仅结构化档案；
- 结构化档案 + 回忆日记；
- 结构化档案 + 向量记忆。

Proment 和悬浮球会明确显示当前组合，不再写“当前只使用向量记忆”。

## 6. 如何验证

部署 R3.2 后发送一条私聊消息，打开悬浮球操作详情：

1. 在“使用资料/Prompt 来源”中找到：
   - `角色档案记忆（结构化档案） · 实际发送`
2. 展开后应看到 `【结构化记忆·按需检索】` 和实际档案条目。
3. 打开“最终原始请求”，搜索：
   - `<structured_archive_memory>`
4. 当前角色选择向量补充但向量为空时，仍应看到结构化档案；不会因为向量为空而整体没有记忆。

## 7. 数据兼容

本版没有修改：

- IndexedDB 表结构；
- 角色、聊天、结构化档案、向量记忆或日记的数据格式；
- 备份文件格式。

不需要重新导入清理版备份，也不需要把设置切回“结构化档案模式”。现有 `memoryMode = vector` 会自动解释为“结构化档案 + 向量补充”。

## 8. 部署提示

若部署后仍看到旧的“三选一”文案或操作中心没有新来源，请清除该站点的缓存与 Service Worker 后重新加载，因为旧静态资源可能仍被浏览器缓存。
