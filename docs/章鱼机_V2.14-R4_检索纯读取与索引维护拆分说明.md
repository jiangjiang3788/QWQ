# 章鱼机 V2.14-R4：检索纯读取与索引维护拆分说明

## 一、版本定位

V2.14-R4 基于 R3 的字段级写入策略，继续收敛“读取记忆”和“修改记忆”的边界。

此前正常聊天在召回结构化记忆时，可能顺便补齐行元数据、更新生命周期、生成并写回行向量，以及累计检索和注入次数。即使这些变化主要是技术元数据，用户没有编辑记忆却发生档案保存，仍会增加竞态、延迟和理解成本。

本版将正常召回改为正式档案纯读取，并把索引和统计移到独立运行态。

## 二、正式档案纯读取

聊天发送前的结构化记忆召回现在只读取正式数据快照：

```text
memoryTables.data 快照
→ 场景与关键词评估
→ 选择候选
→ 组装 Prompt
```

召回过程中不再：

- 给正式行补齐元数据；
- 改写生命周期状态；
- 把向量写入 `row.meta`；
- 增加正式行中的检索或注入次数；
- 因为检索本身生成正式记忆历史。

副作用、反馈和生命周期模块仍可参与判断，但收到的是克隆对象，不能借由召回修改正式档案。

## 三、独立检索运行态

新增：

```text
chat.memoryTables.retrievalRuntime
```

其中保存：

- 行向量索引；
- 索引指纹和建立时间；
- 检索次数与注入次数；
- 最近一次索引维护报告；
- 使用事件幂等标识。

它属于可重建的运行态，不属于用户正式记忆。清除索引不会删除任何记忆内容。

## 四、索引维护流程

行向量不再在普通聊天召回时临时生成。索引只在独立维护动作中建立：

```text
更新索引并预览
或记忆索引维护任务
→ 收集当前可检索行
→ 复用未变化索引
→ 迁移可用旧行向量
→ 为缺失项调用向量 API
→ 写入 retrievalRuntime
```

如果向量 API 未配置或维护失败：

- 正常聊天继续使用关键词检索；
- 不阻断 AI 回复；
- 不修改正式记忆；
- 诊断页显示回退原因和索引覆盖率。

## 五、查询时的混合检索

建立行索引后，混合检索只为本次查询文本生成一个查询向量，不再逐条生成行向量。

没有索引覆盖时，即使设置为“自动”或“混合”，也会安全回退到关键词模式，并提示可执行“更新索引并预览”。

## 六、使用统计

召回和注入次数在最终 Prompt 确定后记录到 `retrievalRuntime.usage`。

统计写入具备事件幂等保护，同一轮重复提交不会重复计数。统计保存失败不会阻断聊天，也不会污染正式记忆内容。

## 七、召回解释

检索诊断页现在显示：

- 是否为纯读取召回；
- 实际使用关键词还是混合检索；
- 索引覆盖数量；
- 每张表的候选数、过滤数和命中数；
- 关键词、语义、标签和反馈分数；
- 每条记忆被选中的原因；
- 最终注入 Prompt 的文本。

按钮统一为“更新索引并预览”，避免把“召回预览”和“建立索引”误解成两个隐藏动作。


## 九、单人版边界

本版不引入后台服务器、常驻 Worker、跨设备索引同步或复杂调度。索引仍保存在当前角色的本地数据中，由用户手动维护或现有本地任务队列触发。

正式数据保持纯读取，不代表整个应用对象绝对不发生任何变化：检索诊断、反馈快照、索引和使用统计仍会写入各自的运行态，但这些变化与正式用户记忆分离。
