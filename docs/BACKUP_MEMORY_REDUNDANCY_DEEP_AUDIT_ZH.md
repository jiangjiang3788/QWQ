# 章鱼喷墨：备份完整性、记忆冗余与代码优化深度审计报告

- 审计日期：2026-07-21
- 代码输入：`QWQ_latest_full_backup_fixed.zip`
- 数据输入：`章鱼喷墨_备份数据_2026-07-21_143428.ee`
- 代码版本标识：`V2.9-R11`
- 数据备份中的应用版本：`6.8.1`
- 审计原则：先复制、后分析；不修改原始文件；优先保留用户内容；只删除可证明为停用、派生、重复或可重建的数据。

## 1. 执行结论

### 1.1 这个备份是否“把所有数据都备份出来了”

从**当前代码的 IndexedDB 结构**看，备份包含了全部 7 张 Dexie 数据表：

`archives`、`characters`、`globalSettings`、`groups`、`myStickers`、`storage`、`worldBooks`。

表数量也与清单一致：2 个角色、67 条全局设置、5 个世界书，其余四张表为空。备份还包含 localStorage，但明确排除了 GitHub Token。

因此，准确结论是：

> 它覆盖了所提供代码当前定义的全部 IndexedDB 表，并备份了大部分 localStorage；但它不是浏览器环境里“任何东西都不遗漏”的物理镜像。

以下内容不属于该备份的完整保证范围：

- GitHub Token（设计上排除）；
- sessionStorage、Cache Storage、Service Worker 缓存；
- 仅以 URL 引用的远程图片、音频或第三方资源的原始文件；
- 浏览器权限、持久化授权和系统级设置；
- 代码版本之外的隐藏/未来数据库表。

另外，代码包标识为 `V2.9-R11`，数据清单中的应用版本为 `6.8.1`。两者的 7 张数据库表结构相符，但版本命名并不一致，所以仍建议在真实浏览器里做一次导入、打开角色、发送消息、检索记忆的冒烟测试。

### 1.2 原始 `.ee` 文件存在恢复完整性缺陷

原备份的 `database/characters.json` 校验值与 ZIP 中实际字节不一致。原因不是内容被篡改，而是旧写入流程中：

1. 校验值按标准 UTF-8 计算；
2. JSZip 写入含 emoji 的 JavaScript 字符串时，把 UTF-16 代理对写成了 CESU-8 风格字节；
3. 导入器再次按标准 UTF-8 计算后，得到不同哈希；
4. 当前严格导入流程会报 `校验失败: database/characters.json`。

因此，原文件虽然可以被 ZIP 工具解开、JSON 也能被兼容解码，但不能称为“在当前导入器下可验证恢复的可靠备份”。优化版已统一为实际 UTF-8 字节写入、按同一字节计算 SHA-256，并增加旧 CESU-8 文件的兼容规范化校验。

### 1.3 冗余的核心不在“结构化记忆正文”

原备份解压后约 101.37 MB，其中 `characters.json` 约 100.01 MB，占约 98.7%。最大角色的主要体积分布如下：

| 数据区域 | 近似体积 | 判断 |
|---|---:|---|
| `memoryTables.taskQueue` | 60.02 MB | 严重异常；递归保存运行态和历史 |
| `unifiedMemory` | 18.20 MB | 所提供代码无引用；旧功能遗留 |
| 聊天 `history` | 16.12 MB | 真实用户聊天为主；保留 |
| `statusSnapshot.replacePattern` | 7.70 MB | 358 份重复模板；可重建 |
| `memoryTables.history` | 3.88 MB | 回滚快照；当前仍有用途，保留 |
| `memoryTables.data` | 0.48 MB | 真正结构化记忆正文；保留 |

也就是说，用户记得的“结构化记忆/档案记忆产生大量数据”这个方向是对的，但精确根因不是结构化记忆条目本身，而是**任务执行过程、调试轨迹、旧系统安全快照和重复渲染模板被长期持久化**。

## 2. 对需求的结构化理解

本次需求可以整理为五个交付目标：

1. **完整性审计**：确认备份包含哪些数据、遗漏哪些数据、是否真的可恢复。
2. **废弃数据识别**：结合代码引用，区分“功能已删除的遗留数据”和“仍在使用的兼容数据”。
3. **数据包瘦身**：在不损失聊天与有效记忆的前提下，生成可校验的精简备份。
4. **产生端修复**：不是只清理一次旧数据，而是阻止任务队列、状态快照和备份编码继续制造冗余。
5. **风险可追溯**：说明删了什么、为什么能删、保留了什么、还需要验证什么。

## 3. 什么叫“深度报告”

这里的深度不是字数多，而是形成完整的证据链：

- 从文件清单、数据库表和校验值判断“备份是否完整”；
- 从数据体积定位异常对象；
- 从异常对象反向追踪到具体写入代码；
- 解释为何会递归膨胀，而不是只说“数据太多”；
- 给出安全删除边界和不可删除边界；
- 实际修改代码并生成清理结果；
- 用校验、语法检查和回归测试证明改动没有破坏现有能力；
- 明确剩余风险，而不是把“测试通过”夸大成“绝对没有问题”。

## 4. 数据审计详情

### 4.1 原始与优化后体积

| 指标 | 原始 | 清理优化后 | 变化 |
|---|---:|---:|---:|
| `.ee` 压缩文件 | 18,820,706 B | 3,575,778 B | 减少 15,244,928 B，约 81.0% |
| `characters.json` 解压字节 | 100,007,003 B | 14,068,776 B | 减少 85,938,227 B，约 85.9% |
| 数据表数量 | 7 | 7 | 不变 |
| 角色数量 | 2 | 2 | 不变 |
| 聊天消息数量 | 19,140 | 19,140 | 不变 |
| 世界书数量 | 5 | 5 | 不变 |

### 4.2 `unifiedMemory`：功能删除后的遗留数据

两个角色都含有顶层 `unifiedMemory`，合计约 18.22 MB。对所提供源码进行全局引用检查，没有发现读取或写入该字段的代码。

大角色内部主要包含：

- `dataSafety.backups`：约 10.57 MB；
- `extractionTraces`：约 6.24 MB；
- `jobs`：约 0.45 MB；
- 以及 retrieval、receipts、entityVersions、reviewDecisions、auditLog、memoryPoints 等旧运行数据。

这些内容属于旧统一记忆系统的安全快照、提取轨迹和审计过程数据；在当前源码中没有恢复入口或消费者。精简包删除了整个 `unifiedMemory`。

同时删除了所提供源码中零引用、且体积很小的角色字段：

`splitBackupMeta`、`chatImageMaxWidth`、`realCameraEnabled`、`saveCallOnInterrupt`、`useRealGallery`。

### 4.3 任务队列递归膨胀：60 MB 的主要根因

问题链路如下：

1. 记忆策略函数生成更新范围，并返回 `info`；
2. `info` 中包含完整 `runtime`（即 `memoryTables`）和完整聊天 `history`；
3. 单表更新结果把整个 `range` 原样返回；
4. 任务队列把完整结果写入 `task.result`；
5. 下一次运行时，`runtime` 已经包含上一次的 `task.result`；
6. 新结果再次保存包含旧结果的完整 runtime，于是形成“快照包含任务，任务又包含快照”的递归式增长。

数据中只有 4 个任务，却占约 60.02 MB：

- 一个成功任务结果约 19.72 MB；
- 一个待审核任务结果约 40.29 MB；
- 另外两个任务体积很小。

第二个任务比第一个更大，正是因为它再次嵌入了已膨胀的任务队列。

#### 已实施修复

- 表更新结果只保留 `range.start` 和 `range.end`，不再返回 `range.info`；
- `task.result` 写库前统一经过白名单化和 32 KB 上限；
- `changedFields` 只保存必要标识，最多 200 项；
- 已完成/取消任务进入历史时移除 payload，只保留紧凑结果；
- 加载旧任务状态时自动清洗已有超大结果；
- 新增回归测试，专门构造带完整 runtime/history 的巨大结果，确认持久化后不再包含 `range.info`。

清理后该角色的 `taskQueue` 从约 60,016,106 B 降至约 4,983 B。待审核与排队任务仍保留，两个已完成任务以紧凑历史形式保留。

### 4.4 状态栏快照重复：约 7.70 MB

358 条消息保存了 `statusSnapshot.replacePattern`，其中 357 份内容完全相同。每份约 21 KB；它本质是渲染模板，不是某条消息独有的数据。

源码只需要快照中的 `regex`，编辑兼容路径还可能需要 `oldRaw`；没有代码读取每条消息里的 `replacePattern`。因此：

- 新消息不再把 `replacePattern` 写入状态快照；
- 旧备份清理时删除该字段；
- `regex` 与必要的 `oldRaw` 保留。

### 4.5 为什么没有删除 `memoryTables.history`

`memoryTables.history` 有 20 份完整记忆数据快照，约 3.88 MB。它确实仍有优化空间，但当前代码把它用于回滚，因此本次保守清理没有删除。

建议下一阶段改为：

- 按字节预算而不是只按条数限制；
- 仅保存变化表或 JSON Patch；
- 相同快照按内容哈希去重；
- 大表使用周期性全量基线 + 增量日志；
- 回滚完成后清理过旧分支。

### 4.6 为什么没有删除聊天消息的 `parts`

大量纯文本消息同时存有 `content` 和等价的 `parts`，约有 18,941 条呈现明显重复。但当前代码多处依赖 `parts` 来表示多模态消息和消息片段，直接删除会带来较高兼容风险。

推荐后续建立明确的消息规范：

- 纯文本只持久化 `content`，读取时按需构造 `parts`；
- 多模态消息持久化 `parts`，`content` 作为派生缓存；
- 给消息结构加 schemaVersion，并通过一次性迁移统一旧记录。

本次未对该区域做激进清理。

## 5. 备份代码优化

### 5.1 修复编码与校验一致性

旧逻辑对“字符串”计算 SHA-256，但交给 JSZip 自行编码，导致实际写入字节与被校验字节可能不同。

新逻辑统一为：

1. `TextEncoder` 生成标准 UTF-8 `Uint8Array`；
2. 对该字节数组计算 SHA-256；
3. 把同一个字节数组写入 ZIP；
4. 导入时读取原始字节后校验，再解码 JSON。

同时增加旧 CESU-8 代理对规范化兼容；只有规范化后的字节哈希与清单完全一致时才接受，避免为了兼容而绕过完整性校验。

### 5.2 增加两种备份模式

- **精简备份（推荐）**：保留业务数据，排除已确认的停用/派生冗余；GitHub 自动备份默认使用该模式。
- **原始完整备份**：保留数据库当前全部字段，用于法证或需要逐字节保留旧状态的场景。

两者使用同一 `ovo-full-backup v1` 容器和恢复流程，清单中的 `mode` 用于区分。

### 5.3 扩展显式清理入口

“清除冗余/无用数据”操作现在会在用户确认后清理：

- `unifiedMemory` 旧数据；
- 重复的状态栏模板；
- 已完成任务的 payload 与递归结果；
- 原有的空角色、空群聊、未引用世界书和无效表情数据。

操作前提示明确说明：聊天正文、结构化记忆正文、待审核/排队任务和回滚历史会保留。

## 9. 剩余风险与建议优先级

### P0：实际浏览器恢复冒烟测试

自动校验能证明文件结构、哈希和代码契约成立，但不能完全替代浏览器中的 IndexedDB 写入和 UI 行为。发布前应在隔离浏览器配置中完成：

1. 导入精简包；
2. 打开两个角色；
3. 浏览历史消息；
4. 打开结构化记忆工作区；
5. 检查待审核任务；
6. 发送一轮消息；
7. 再次导出精简备份并重新导入。

### P1：为持久化对象建立 schema 与预算

每个可增长容器应同时有：

- schemaVersion；
- 最大条数；
- 最大字节数；
- 单条最大字节数；
- 过期策略；
- 可重建字段标记；
- 迁移与清理函数。

仅用“最多 N 条”无法防止一条对象包含几十 MB。

### P1：回滚快照改增量

`memoryTables.history` 是下一个最值得优化的区域。建议使用全量基线 + 增量变更，并设置例如 8–16 MB 的总预算。

### P2：消息 `content/parts` 规范化

这是潜在的数 MB 级进一步空间，但需要 schema 迁移和多模态回归测试，不建议在本次保守清理中直接删除。

### P2：加入体积可观测性

建议在开发/设置页显示：

- 每个角色总字节；
- 聊天、记忆正文、任务、快照、媒体分别占用；
- 最近一次增长来源；
- 导出前预计体积；
- 一键生成数据健康报告。

## 10. 最终判断

用户的核心意图可以概括为：

> 不是简单把压缩包变小，而是确认备份是否可信，找出已删除功能和现行功能产生的冗余，保住真正的聊天与记忆内容，修复产生冗余的代码，并留下可以验证、可以继续维护的结果。

本次结果实现了这个目标的主体：原始文件保持不动；提供了校验可通过的精简数据包；修复了任务递归膨胀、状态模板重复和备份编码不一致；统一了模板与构建清单；并给出了仍需浏览器冒烟测试和下一阶段架构优化的明确边界。
