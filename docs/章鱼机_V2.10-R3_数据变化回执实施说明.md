# 章鱼机 V2.10-R3 数据变化回执实施说明

## 1. 本版目标

V2.10-R0 让用户看见“正在执行什么”；R1 让用户看见“发送了什么以及内容来自哪里”；R2 让用户看见“回复后又触发了哪些后台工作”；R2.1 修复了详情截断、层层套框、消息时间、审核交互和重复发送疑问。

V2.10-R3 继续回答最后一个核心问题：

> 这次操作最终在应用里新增、更新、删除或提出了什么数据变化？

本版引入轻量的 Mutation Receipt（数据变化回执），把真实写入与仅供审核的建议分开显示，同时避免保存完整数据库快照。

## 2. 用户可见变化

悬浮球操作详情新增“数据变化”区域，可展示：

- 新增了几条聊天消息；
- 哪些角色档案字段被更新，以及必要的修改前/修改后内容；
- 结构化记忆整理生成了多少待确认建议；
- 用户点击“保存已接受项”后，哪些结构化记忆真正新增、更新或删除；
- 是否生成了新日记；
- 是否新增了向量记忆；
- 是否保存了小剧场及其通知消息；
- 这些变化来自主操作还是某个后台子任务。

操作卡片会显示轻量汇总，例如：

```text
数据变化 7 项 · 新增 4 · 更新 1 · 待确认 2
```

详情中每条回执会显示动作、对象类型、摘要、发生时间和关联 ID。必要时可展开查看写入内容或前后差异。

## 3. “待确认”与“已经写入”的区别

结构化记忆链路被明确拆成两个阶段：

1. **生成待确认建议**：AI 整理出候选，但尚未改动正式档案；
2. **保存已接受项**：用户选择并提交后，才记录为真实新增、更新或删除。

因此操作中心不会再把“AI 建议了三项”误报为“数据库已经写入三项”。

## 4. 新增的统一数据变化协议

`OVOOperationRuntime` 新增：

```js
recordMutation(operationId, mutation)
recordMutations(operationId, mutations)
```

单条变化采用受控结构：

```js
{
  action: 'create | update | delete | pending',
  entityType: 'chat_message | character_memory | structured_memory | memory_review | journal | vector_memory | theater',
  entityId: '关联对象 ID',
  title: '用户可读标题',
  summary: '简短结果说明',
  count: 1,
  before: '必要的修改前内容',
  after: '必要的写入后内容',
  fields: ['涉及字段'],
  at: 'ISO 时间'
}
```

## 5. 存储控制

为防止操作中心本身成为新的冗余源：

- 单次操作最多保存 80 条变化回执；
- 单个前值或后值最多保存约 4,000 字符；
- 父操作只保存自己的变化和子操作 ID；
- 父操作的变化数量通过子操作摘要计算，不复制子操作正文；
- 较旧历史记录会去除详细前后值，仅保留轻量摘要；
- 不保存完整角色对象、聊天数组、结构化记忆表或任务队列。

## 6. 已接入的真实写入点

### 6.1 聊天回复

角色回复解析并写入聊天后，按照实际新增消息 ID 记录变化。组合回复被拆成多条聊天气泡时，会记录真实新增条数，但不会因此认为模型请求发送了多次。

### 6.2 回复内角色档案更新

Sidecar 实际应用档案字段后，记录更新字段、摘要和必要的前后值。被拒绝或未产生变化的指令不会伪装成写入。

### 6.3 自动日记

日记生成完成并写入后，记录新日记 ID、标题、消息范围和内容摘要。

### 6.4 向量记忆

向量记忆生成完成并保存后，记录新增条目及其关联 ID，不保存向量数组本身。

### 6.5 小剧场

手动与角色主动小剧场保存成功后，记录小剧场对象；角色主动创作同时产生通知/分享消息时，也会记录该聊天消息。

### 6.6 结构化记忆

- 直接应用的变更记录为真实新增、更新或删除；
- 审核草案记录为 `pending`；
- 用户保存已接受项时，创建独立的“保存结构化档案审核结果”操作，并逐项记录实际写入。

## 7. 父子操作汇总

主聊天操作可以汇总后台子任务的数据变化：

```text
生成角色回复
├─ 新增聊天消息 2 条
├─ 更新角色档案 1 项
├─ 生成待确认记忆 3 项
├─ 新增角色日记 1 篇
└─ 新增向量记忆 1 条
```

父记录只保存汇总计数。打开某条子任务后，才能查看它自己的详细回执。

## 8. UI 原则

本版继续遵守 R2.1 的产品约束：

- 扁平详情，不制造层层小框；
- 变化列表不放进固定高度小滚动区；
- 默认讲人话，高级信息按需展开；
- 明确区分“待确认”和“已经写入”；
- 时间、关联对象和来源操作可追踪；
- 原始 Prompt、来源视图和数据变化互不混淆。

## 9. 兼容性

- 不修改 IndexedDB 表结构；
- 不修改角色、聊天、世界书、日记或结构化记忆的数据格式；
- 不替换现有 `OvoMemory`、MemoryTable、日记、向量记忆和小剧场模块；
- R0、R1、R2、R2.1 的操作记录可继续读取；
- 旧操作没有变化回执时，界面会明确显示“本次操作没有记录到持久化数据变化”。

## 10. 测试范围

本版执行：

- 全量 JavaScript 语法检查；

- V2.1—V2.9-R11 历史回归；

- 发布包文件清单、语法和二次解压回归。

## 11. 部署后重点实测

由于测试环境没有使用用户的真实外部模型 API，部署后建议依次实测：

1. 发送一次会拆成多条气泡的角色回复，确认显示真实新增消息条数且模型请求仍为一次；
2. 触发一次回复内档案更新，确认前后值正确；
3. 生成结构化记忆待处理项，确认先显示“待确认”；
4. 选择并保存审核项，确认出现独立保存操作及真实写入回执；
5. 触发一次自动日记、向量记忆或自动小剧场，确认变化挂在正确父操作下。

## 12. 下一阶段

按既定路线，V2.10-R4 应完成全功能登记与覆盖审计：找出仍绕过统一运行时的 AI 调用和持久化写入，使图片识别、头像、表情包、通话、电量互动、拉黑判断等功能也能稳定产生完整操作回执。
