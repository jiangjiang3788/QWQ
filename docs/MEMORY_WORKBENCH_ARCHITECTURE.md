# 记忆 UI 工作台（V2.9-R7）

## 用户入口

结构记忆 App 保留三个一级入口：

- 记忆：当前状态、活跃待办、结构化记忆内容，以及高频的“更新与整理”。
- 待处理：更新审核、短期候选、可靠性复核、使用反馈和失败任务。
- 管理：模板、召回诊断、质量测试、历史和导入导出。

旧的技术视图仍保留在隐藏兼容容器中，不参与普通 UI 展示。旧的更新设置路由已经删除；“更新与整理”只有记忆页这一处正式入口。

## 为什么“更新与整理”属于记忆页

更新、游标和表级调度是用户查看记忆后最常执行的下一步，不是低频系统管理。R6 将该面板直接展开在记忆页，同时保留管理页负责模板和诊断，降低高频操作路径长度。

## 自动整理的四种通道

每张表都拥有独立的运行态 `automationMode`：

- `sidecar`：由正常聊天同一次请求维护，不额外调用提取 API。适用于当前状态、待办、近期经历和日常观察。
- `engine`：跟随角色级全局轮数/消息数设置。默认用于中期总结。
- `table`：使用模板中该表自己的更新频率和窗口。
- `manual`：只允许用户手动更新。

模板定义保存“默认策略”，角色绑定中的 `tableStates` 保存运行态覆盖、游标和执行状态。两者不能混为同一层数据。

## R5 自动更新失效的根因

旧链路需要同时满足：

```text
全局自动更新开启
AND 表级 updatePolicy.enabled = true
AND 表级 triggerMode != manual
AND 表运行态 enabled = true
AND 达到轮数或消息数阈值
```

V2.8 默认模板为了保留“一轮聊天一次主请求”，四张短期表被设为 sidecar/手动；中期和长期表也默认手动。因此即使页面显示全局自动更新已开启，调度器仍会过滤掉全部表。

R6 不再用一个布尔值模糊表达两种更新机制，而是先确定表的自动化通道，再解析有效策略。界面同时显示可自动执行的表数、已到期表数和每张表的当前通道。

## 游标原则

- 新建运行态默认从最新消息开始，避免升级后突然重放全部历史。
- 用户切换到自动模式时不静默重置游标。
- 需要整理历史积压时，用户通过自定义游标明确选择起点。
- 草案待审核期间不推进游标；应用或整批拒绝后才推进。

## UI 与模块边界

`workspace.js` 负责：

- 一级入口与内部视图映射；
- 待处理数量；
- 当前状态与活跃待办摘要；
- 待处理和管理概览卡片；
- 详情页标题与返回路径。

`memory_table.js` 负责：

- 数据读取和页面总调度；
- 更新与整理面板；
- 表级自动化通道展示和交互；
- 旧视图渲染器的兼容调用；
- 原事件处理和业务服务连接。

`memory_table_policy.js` 负责：

- 全局引擎、模板策略和运行态覆盖的归一化；
- 表级自动化通道推断；
- 是否到期、处理范围、轮次和游标计算；
- 不包含 DOM 操作。

## 层级约束

- 一级：记忆 / 待处理 / 管理。
- 二级：概览卡片进入单个功能页。
- 临时层：编辑、确认和导入弹窗。

不增加三级标签或嵌套折叠导航。高级设置只保留一层 `<details>`。

## V2.9-R10：工作区状态入口

`selectMemoryWorkspace(workspace, view)` 是“记忆 / 待处理 / 管理”状态迁移的唯一控制器。顶部标签、工作区卡片、详情返回及外部 facade 均不得直接修改 `uiState.workspace` 与 `uiState.tab`。

## V2.9-R11：工作区状态所有权

工作区状态遵循单向优先级：显式导航目标 > 当前角色已水合 UI > 角色 runtime > 默认值。

- `getCurrentMemoryTableChat()` 只在首次进入或角色切换时水合 UI，不再在每次读取时覆盖界面状态。
- `applyMemoryWorkspaceState()` 是 workspace/view 的唯一提交入口，同时更新 `uiState` 与角色 runtime。
- `selectMemoryWorkspace()` 必须先解析当前角色，再提交目标状态，避免旧 runtime 反向覆盖。
- 基础工作区导航通过独立幂等绑定完成，不依赖审核、任务、质量等可选业务控件初始化。
