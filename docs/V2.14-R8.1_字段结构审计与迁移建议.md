# V2.14-R8.1 字段结构审计与迁移建议

## 审计范围

基于用户提供的单人精简备份进行只读审计：

```text
模板：1
表格：8
字段：94
正式多行记忆：279
```

未发现重复表 ID、重复字段 ID、同表重复字段名或指向不存在字段的正式单元格。问题主要不是“数据缺失”，而是表职责和字段职责发生了历史叠加。

本文件只提出迁移建议，R8.1 不自动执行任何破坏性结构调整。

## 一、待办与近期经历复制了同一套 14 字段

两张表当前字段完全相同：

```text
事件ID、创建时间、最后更新时间、完成时间、类型、标题、内容、
相关主体、影响、当前状态、后续待办、结果、搁置或取消原因、原始记录ID
```

实际数据表明：

- 待办 49 行，状态确实包含待办、进行中和已完成；
- 近期经历 31 行，31/31 的当前状态都是“已完成”；
- 近期经历的后续待办和搁置原因大多为空。

建议拆分为：

### 待办

```text
标题、内容、状态、类型、截止时间或触发条件、优先级、
相关主体、下一步、结果或关闭原因
```

当前结构缺少真正有用的“截止时间/触发条件”和“优先级”。

### 近期经历

```text
发生时间、事件类型、标题、内容、相关主体、影响或意义、来源
```

不再需要完成时间、当前状态、后续待办和搁置原因等任务工作流字段。

## 二、技术字段与 row.meta 重复

多个表仍把以下技术信息作为用户可见字段：

```text
事件ID、原始记录ID、创建时间、最后更新时间、完成时间、
置信度、数据完整度、来源说明、审核状态、确认状态
```

R8 已经提供：

```text
row.id
meta.identity.recordKey
meta.identity.sourceRefs
meta.identity.sourceFingerprint
firstSeenAt / lastSeenAt
WorkItem 状态
写入回执与历史
```

建议迁移顺序：

1. 先在 UI 中把技术列标记为“高级/隐藏”；
2. 将来源和身份迁入 `row.meta.identity`；
3. 验证召回、导入导出和历史引用没有生产依赖；
4. 最后才从模板中删除旧技术字段。

不建议在一次版本中直接删除这些列。

## 三、核心确认档案混入角色设定

核心档案的 12 个字段中包含多项 `char_*`：

```text
char_人格底色、char_核心能力、char_回应原则、char_喜好与生活习惯、
char_边界与底线、char_弱点、char_秘密
```

这些字段已经配置为 `runtime_only`，说明其主体不是用户正式事实。建议最终分离：

### 用户与关系确认档案

```text
user_人格底色
双方_核心关系定义
双方_称呼系统
双方_相处公约
```

### 角色人设或角色运行配置

所有 `char_*` 字段。

R8.1 已停止把旧 `runtime_only` 正式值注入记忆上下文，但旧值仍保留，等待未来可回滚迁移。

## 四、当前状态表职责过重

当前状态表有 17 个字段，同时承担：

- 用户明确短期状态；
- 模型推断状态；
- 角色当前状态；
- 角色对用户的判断；
- 回应策略和边界提醒。

建议最终拆分：

### 用户当前状态

```text
状态记录时间、当前场景、身体状态、精神状态、压力源、当前需求、
体力、精力、当前风险、状态有效期
```

### 角色会话运行态

```text
角色当前场景、角色精神状态、角色运行状态、对用户判断、
回应策略、边界提醒、下一步建议
```

R8.1 已先完成数据路由：明确事实可正式写，模型推断进入运行态。UI 和模板结构拆分留给后续迁移版本。

## 五、日常观察字段需要按“自述”和“推断”拆分

当前字段：

```text
日期、睡眠情况、饮水情况、运动与活动、身体状态、
精力与情绪、数据完整度、来源说明
```

问题：

- “精力与情绪”可能同时表示用户自述、模型推断和系统日总结；
- “数据完整度”和“来源说明”属于元数据；
- 日期曾出现带时间段描述的非标准值；
- 规则要求同日合并，但旧通用 RecordIdentity 没有按日期强 Upsert。

建议：

```text
日期
时间段（可选）
睡眠自述
饮水自述
运动与活动
身体状态自述
精力自述
情绪自述
```

模型推断单独进入运行态，不与用户自述共用一个正式字段。数据完整度和来源迁入 meta。

R8.1 已先实现按日期 Upsert，避免继续新增同日重复行。

## 六、中期总结混合三种记录类型

当前 12 行记录类型包括：

```text
成长事件：9
更新标记：2
周期摘要：1
```

三种类型所需字段并不相同。当前“内容或摘要”仅 1/12 有值，而旧模式、新反应、具体证据等字段主要只适用于成长事件。

建议拆成：

### 成长经验

```text
发生时间、主题、旧模式、新反应、具体证据、成长意义、
稳定程度、可复用经验
```

### 阶段摘要

```text
周期、主题、摘要、主要变化、未解决事项、下一阶段关注点
```

“更新标记”应进入写入历史或维护日志，不作为正式记忆行。

另外，当前置信度主要使用 `5` 和 `0`，而系统其他位置使用 0–100，需要在未来 Schema 迁移中统一量纲。

## 七、长期候选审核状态与 WorkItem 重复

长期候选表当前为 0 行，字段中仍包含“审核状态”。R8 已经建立统一 WorkItem，审核状态继续作为普通字段会形成两套真相来源。

建议长期候选正式内容只保留：

```text
候选类别、候选内容、支持证据、反例或例外、观察跨度、
独立证据次数、置信度
```

批准、拒绝和等待处理状态由 WorkItem 和晋升历史管理。

## 八、稳定长期库可以收敛

稳定长期库当前 179 行、8 个字段：

```text
来源域、维度或类型、分类、内容、原置信度、确认状态、
例外或适用场景、原始记录ID
```

实际数据：

- 原置信度 179/179 都是 `5`；
- 确认状态 179/179 都是“迁移已确认”；
- 例外或适用场景 179/179 为空；
- 原始记录 ID 与新身份元数据重复。

建议用户可见结构收敛为：

```text
类型
分类或主题
内容
适用场景或例外
```

来源、确认状态、旧置信度和原始记录引用迁入 meta 与变化链。

## 九、推荐迁移顺序

不建议在 R8.1 直接改表。后续可按以下顺序实施：

1. **只读映射预览**：显示每个旧字段将迁往何处；
2. **完整角色备份**：迁移前强制生成；
3. **新增目标结构**：不覆盖旧表；
4. **双写或副本迁移验证**：检查行数、来源和内容哈希；
5. **切换读取入口**：确认新结构行为正确；
6. **隐藏旧技术字段**：先隐藏，不删除；
7. **观察一个稳定版本周期**；
8. **最后删除零引用旧字段**。

推荐将真正的字段结构迁移放在 V2.15-R0 的“旧兼容字段和技术入口清理”阶段，与浏览器 E2E 和完整备份恢复测试一起完成，而不是塞入 R9 生命周期版本。
