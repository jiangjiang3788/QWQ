# 导航与 UI 信息架构（V2.9-R6）

## 问题定义

本轮不是单一的“样式问题”，而是五类问题叠加：

1. **视图状态同步缺陷**：点击后的 UI 状态被持久化状态重新覆盖，表现为“待处理点不开”。
2. **信息架构重复**：角色、联系人、聊天、记忆、设置在桌面、Dock、底栏和设置中重复出现。
3. **导航栈碎片化**：不同页面分别写死返回目标，设置页和角色 App 无法回到真实来源。
4. **交互模型不一致**：桌面像长网页，内部页面又像手机 App，导致滚动和返回预期不统一。
5. **可观测性不足**：出现异常时只能看零散 console，无法快速确认导航、记忆、设置和当前角色状态。

统一称为：**信息架构、导航状态与交互一致性问题**。

## 单一入口原则

### 首页 Dock

只放四个最高频入口：

```text
聊天 / 角色 / 记忆 / 设置
```

这四个入口不再同时出现在桌面网格中。

### 桌面 App 页

只放独立任务型能力：

```text
世界书 / 剧场 / 收藏 / 提醒 / 搜索
```

使用原生横向滚动和 CSS Scroll Snap，后续增加 App 时自动扩展为多页，不使用长纵向列表。

### 联系人

联系人属于聊天域，保留在聊天底栏中，不再作为桌面重复 App。

### 设置

设置只承载“配置”，不重复承载“操作入口”。

```text
个人与角色：我的档案、当前角色设置
模型与能力：API 与模型、高级规则
外观与桌面：外观、壁纸、图标、状态栏
数据与系统：备份、迁移、诊断
```

角色列表、角色记忆等日常操作分别回到角色 App 和记忆 App。

## 角色 App

每个角色只保留一张卡片和三项核心动作：

```text
聊天 / 记忆 / 设置
```

角色 App 不再复制联系人页，也不再自己维护一套返回逻辑。

## 记忆 App

顶部固定为三个工作区：

```text
记忆：查看和编辑档案
待处理：更新确认、短期候选、冲突复核、反馈、失败任务
管理：模板、召回、质量、历史和更新策略
```

工作区状态写入统一通过 `persistWorkspaceState()`。只有切换角色或显式要求时，才从角色运行时状态重新装载 UI，避免渲染过程覆盖用户刚刚点击的状态。

## 导航栈

所有页面切换统一经过：

```js
OvoNavigation.go(target)
OvoNavigation.back(fallback)
OvoNavigation.reset(target)
OvoNavigation.snapshot()
```

返回按钮不再直接跳固定页面，而是优先返回真实来源；没有历史时才使用按钮的 `data-target` 作为兜底。

角色上下文在角色 App、设置和记忆之间保留，只在回到首页、聊天列表或联系人主页面时清空当前会话。

## 诊断入口

设置 → 数据与系统 → 运行状态。

控制台也可以使用：

```js
OvoDiagnostics.snapshot()
OvoDiagnostics.report()
OvoDiagnostics.log()
OvoDiagnostics.open()
```

报告包含：

- 当前页面和返回栈；
- 角色、群聊和记忆模板数量；
- 当前角色及记忆工作区；
- 记忆、设置和角色设置健康检查；
- 最近的脚本错误和未处理 Promise 错误。

报告不包含 API Key。
