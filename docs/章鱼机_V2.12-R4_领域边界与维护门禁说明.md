# 章鱼机 V2.12-R4：领域边界与维护门禁说明

## 一、本版定位

V2.12-R4 是 V2.12 收敛计划的工程化收口版本。本版不增加新的用户页面、卡片或操作入口，而是把已经完成的五条记忆能力正式固定为可维护的领域：

- 记忆表格；
- 记忆治理；
- 记忆引用与作用；
- 表结构编辑；
- 模型整理与跨表上下文。

目标是阻止后续开发再次把具体模块直接堆回 `memory_table.js`，并用自动检查约束代码体积、模块依赖和 UI 密度。

## 二、主要改动

### 1. 七个公开领域门面

新增：

```text
js/features/memory/domains/
├─ platform.js
├─ foundation.js
├─ tables.js
├─ governance.js
├─ retrieval.js
├─ schema.js
└─ update.js
```

主控制器以前会直接读取策略、标签、关系、侧栏、表格、缓存、编辑器、跨表上下文等大量叶子模块。

现在只依赖：

```text
memoryPlatformDomain
memoryFoundationDomain
memoryTablesDomain
memoryGovernanceDomain
memoryRetrievalDomain
memorySchemaDomain
memoryUpdateDomain
memoryArchitecture
```

因此控制器直接叶子模块依赖从三十余项下降为 **0**。模块内部实现可以继续替换，只要公开门面保持稳定，页面控制器不需要同步修改。

### 2. 领域归属契约

新增：

```text
architecture/memory_domains.json
```

它登记：

- 每个领域门面的文件位置；
- 每个叶子模块归属哪个领域；
- 主控制器允许依赖哪些公开门面；
- 关键文件的代码行数上限。

当前共有：

- 7 个领域门面；
- 43 个明确归属的叶子模块；
- 0 个重复归属模块。

### 3. 运行时架构健康检查

新增：

```text
js/features/memory/architecture.js
```

页面加载时，主控制器会先执行：

```js
Kernel.require('memoryArchitecture').assertHealthy();
```

如果某个领域门面未加载、加载顺序错误或缺少模块，记忆控制器不会在不完整状态下继续运行。

### 4. UI 维护预算

新增：

```text
architecture/ui_budgets.json
js/features/memory/maintenance.js
```

运行时可检查：

- 当前虚拟表可见数据行；
- 活动编辑器数量；
- 共享行菜单数量；
- 常驻编辑按钮数量；
- 页面级横向溢出。

当前表格预算：

| 项目 | 上限 |
|---|---:|
| 可见数据行 | 30 |
| 活动编辑器 | 1 |
| 共享行菜单 | 1 |
| 常驻行编辑按钮 | 0 |
| 页面级横向溢出 | 1px |

这会防止后续改动重新出现“179 行全部渲染”“每行一个编辑按钮”“页面被表格撑宽”等问题。

### 5. 静态架构门禁

新增：

```text
tools/check_memory_architecture.py
```

检查内容包括：

1. 领域门面是否存在并正确注册；
2. 叶子模块是否被门面实际暴露；
3. 是否存在重复模块归属；
4. 主控制器是否绕过门面直接依赖叶子模块；
5. 脚本加载顺序是否正确；
6. 关键文件是否超过行数预算；
7. 已退役的悬浮球二级页面和拆分模板编辑器是否重新出现。

该门禁已接入原有 `tests/run_static_checks.py`，不是一份孤立报告。

## 六、兼容性

本版没有修改：

- 记忆 schema；
- `.ee` 备份格式；
- 8 张表及字段定义；
- 标签、关系、生命周期和审核数据；
- 模型整理 Prompt；
- 召回与反馈协议；
- 悬浮球用户操作；
- 表结构编辑器用户操作。

因此它是工程架构升级，不是数据迁移版本。

## 七、后续开发规则

后续新需求应遵循：

1. 先确定所属领域；
2. 领域内复用现有服务；
3. 跨领域能力通过公开门面暴露；
4. 页面控制器禁止直接依赖叶子模块；
5. 新 UI 必须通过预算检查；
6. 超过行数预算时先拆分，再增加能力。

这套规则用于确保系统不会再次回到“一个文件知道所有模块、一个页面堆满所有操作”的状态。
