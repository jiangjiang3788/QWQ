# 章鱼机代码与产品深度评审（V2.9-R6）

## 项目定位

章鱼机是一个本地优先、单页 PWA 形态的“角色关系与长期记忆工作台”。表面是仿手机/微信聊天，核心不是普通聊天 UI，而是围绕角色持续维护：

- 角色档案和关系上下文；
- 私聊、群聊、世界书和创作能力；
- 分层结构化记忆；
- 记忆提取、审核、召回、生命周期和质量反馈；
- 多 API、外观、数据备份和高级行为配置。

数据主体保存在 IndexedDB/Dexie；AI 请求通过用户配置的 API；界面由经典脚本、全局状态和动态 DOM 共同驱动。

## 记忆系统现状

现有结构化记忆已经不只是“表格编辑器”，而是一条完整流水线：

```text
聊天消息
→ 同请求 sidecar 提取当前状态/待办/近期候选
→ 更新草案与审核
→ 表格写入
→ 标签、场景和相关性召回
→ 注入聊天上下文
→ 使用反馈
→ 冲突、过期、可靠性和质量检查
```

模块边界包括：kernel、domain、API adapter、workspace、policy、review、retrieval、sidecar、effects、lifecycle、tasks、feedback、quality 和 facade。

用户真正需要的不是看见所有模块，而是三件事：

1. 档案能持续更新且不会乱写；
2. 能看懂“系统记住了什么、为什么记住、哪里需要确认”；
3. 出问题时能定位是 API、提取、审核、召回、注入还是 UI 状态。

## 当前代码优势

- 记忆 schema、原始数据和 209 条迁移记录有回归保护；
- R1～R5 已逐步拆出记忆内核、记忆工作台、App Registry、设置组件和角色设置控制器；
- 旧公共函数仍保留兼容门面，降低大规模重构风险；
- 测试已覆盖审核、检索、sidecar、副作用、生命周期、任务、反馈和质量。

## 当前主要技术债

### 1. 全局状态仍然过多

`db`、`currentChatId`、`currentChatType`、大量全局函数和动态 DOM 互相依赖。任何页面都可能修改当前角色或屏幕状态，容易出现“看似 UI，实为状态覆盖”的问题。

建议逐步形成三个明确状态域：

```text
SessionState：当前角色、会话、页面、导航栈
DataStore：IndexedDB 数据读写与迁移
FeatureRuntime：记忆、设置、TTS 等模块运行状态
```

### 2. 业务逻辑与视图仍有交叉

部分模块同时负责数据修改、HTML 拼接、事件绑定和持久化。下一阶段应优先拆“命令”和“查询”，而不是继续按文件行数机械拆分。

推荐形态：

```text
MemoryCommands.acceptReview(...)
MemoryQueries.getInboxCounts(...)
MemoryViews.renderInbox(...)
```

### 3. 页面标识和返回规则缺少正式路由模型

R6 增加了导航栈，但屏幕仍以 DOM id 为路由。下一步可增加路由描述表，定义：页面、所属 App、是否需要角色、默认返回和进入钩子。

### 4. 诊断仍偏开发者

R6 已建立统一快照。下一步应把诊断结果翻译成用户能执行的结论，例如：

```text
记忆 API 未配置
最近一次提取失败
有 3 条审核草案未处理
当前角色没有绑定模板
召回结果为空，但聊天历史正常
```

### 5. 缺少真实移动端端到端测试

静态和 Node 回归能保护结构，但不能完全覆盖触摸、软键盘、WebView 安全区、滚动和 Android 返回键。发布前应建立至少一条真实设备验收路径。

## 推荐后续优先级

### P0：让档案记忆真正“可用”

- 增加角色记忆首页状态摘要：模板是否绑定、最近提取、待审核、最近召回、最近失败；
- 给每次记忆更新建立可追踪的 round id；
- 从聊天消息追到 sidecar、审核草案、最终表格行和注入记录；
- 提供一键“为什么这条没记住/为什么这条被发送”。

### P1：让控制台和诊断“好用”

- 统一日志分类：navigation、memory.extract、memory.review、memory.retrieve、memory.inject、api、storage；
- 所有异步任务记录开始、成功、失败、耗时和角色 id；
- 诊断报告支持导出脱敏 JSON；
- 增加最近一次 AI 请求的模型、用途、消息数量和错误，不记录密钥和完整隐私内容。

### P2：提高可维护性

- 新功能只能通过 App Registry、Navigation、Settings facade 和 Memory facade 进入；
- 禁止新代码直接从多个页面写 `currentChatId`；
- 为核心命令增加纯函数测试；
- 建立 DOM id、数据库字段和公共 API 的兼容清单；
- 每个版本必须通过全量语法、静态资源、记忆数据哈希和专项回归。

### P3：继续优化 UI

- 聊天域只保留聊天、联系人和首页三个底栏入口；
- 角色、记忆、设置是首页一级 App；
- 高级功能默认收进设置或“管理”；
- 页面只允许一种主要滚动方向；
- 顶部返回、标题、右侧动作统一为同一 App Header 组件。

## 用户体验验收标准

一个普通用户应该能够在不理解“sidecar、retrieval、lifecycle”等术语的情况下完成：

1. 创建角色并开始聊天；
2. 打开记忆，看到当前档案和待处理数量；
3. 确认或拒绝一次更新；
4. 查看某条记忆何时写入、是否被聊天使用；
5. 在设置里完成 API、外观和备份；
6. 返回时永远回到刚才来的页面；
7. 出错时打开运行状态，得到明确的检查结论。
