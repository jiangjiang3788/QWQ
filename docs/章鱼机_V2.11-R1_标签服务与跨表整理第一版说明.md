# 章鱼机 V2.11-R1：标签服务与跨表整理第一版

## 1. 本版目标

V2.11-R1 延续 R0 的扁平表格 UI，不增加新的外层卡片或常驻设置区。本版集中完成两项底层能力：

1. 让结构化记忆整理请求由大模型直接生成统一标签；
2. 整理某一张表时，按表格职责检索并携带其他相关记忆表的相关行。

本版同时开始从 `memory_table.js` 中抽离整理职责，降低主控制器继续膨胀的风险。

## 2. 新增模块

### `tag_service.js`

统一处理现有标签结构：

```js
row.meta.tagBundle = {
  topic: [],
  scene: [],
  entity: [],
  effect: ''
}
```

没有新增 `aiTags`、`generatedTags` 或其他平行字段。

能力包括：

- 标签标准化与数量限制；
- 解析模型返回的 `<tags>`；
- 写回现有 `tagBundle`；
- 标签变化比较；
- 标签更新后清空旧检索向量，避免向量与文本不一致；
- 为模型整理请求提供统一标签输出协议。

`effect` 继续使用已有记忆作用类型：

```text
fact
temporary_state
soft_preference
hard_boundary
reminder
historical_context
candidate
```

### `context_assembler.js`

新增只读跨表上下文组装器。它会识别表格职责：

```text
core        核心确认档案
current     当前状态
todo        待办与承诺
event       近期经历
observation 日常观察
medium      中期总结
candidate   长期候选
long        稳定长期特征
```

整理目标表时，不会把全部记忆无差别塞入 Prompt，而是：

1. 根据目标表选择允许关联的表；
2. 使用现有 `MemoryPolicy.selectRelevantItems()` 检索相关行；
3. 保留表 ID、行 ID、字段名和已有标签；
4. 按字符预算裁剪；
5. 以 `<related_memory_tables>` 只读结构加入同一次整理请求。

相关表只用于判断重复、冲突、前后关系和阶段变化，模型不能直接修改非目标表。

### `update_service.js`

从 `memory_table.js` 中抽出：

- 消息范围收集；
- 时间戳和聊天文本构建；
- 目标表定义构建；
- 候选行选择；
- 最终整理 Prompt 组装。

主控制器继续负责 UI、任务和审核调度，具体 Prompt 组装由独立服务负责。

## 3. 模型整理协议

对于 rows 表的新增行和有实质变化的更新行，模型必须输出：

```xml
<row op="add">
  <field fieldId="字段ID">值</field>
  <tags
    topic="主题1,主题2"
    scene="场景1"
    entity="主体1"
    effect="historical_context"
  />
</row>
```

更新现有行时必须引用现有 `rowId`：

```xml
<row op="update" rowId="现有行ID">
  <field fieldId="字段ID">新值</field>
  <tags ... />
</row>
```

协议明确要求：

- 先检查目标表和相关表；
- 优先更新或补充现有记录；
- 不为同一事实反复新增；
- 标签来自本行内容，不机械复制相关表标签；
- topic、scene、entity 均有数量限制；
- 无变化时不输出。

## 4. 审核与写入

标签已经进入现有审核链路：

- 新增行审核时包含建议标签；
- 更新行审核时包含标签变化；
- 接受审核后写入同一行的 `meta.tagBundle`；
- 直接应用模式也使用相同服务；
- 数据变化回执可以显示标签对象变化。

## 5. 数据体积控制

跨表上下文只在当前整理请求中临时组装。

任务和角色数据中只保存摘要：

- 相关表数量；
- 相关行数量；
- 上下文字符数；
- 表名、角色和选择原因。

不会把完整 `<related_memory_tables>`、完整角色对象或完整聊天重新保存进任务结果。

任务输入体积估算已经计入相关表上下文，便于队列正确评估请求大小。

## 6. 真实数据运行结果

使用用户提供的清理优化版 `.ee`：

- 角色：阿沉；
- 聊天消息：19,133 条；
- 模板：分层可检索角色记忆 V2（含原数据）；
- 表格：8 张。

以“中期总结与成长经验”为目标表，实际装配：

| 相关表 | 行数 | 作用 |
|---|---:|---|
| 近期经历、想法与重要事件 | 2 | 核对近期事件脉络 |
| 待办、承诺与未完成事项 | 3 | 核对承诺和行为变化 |
| 稳定长期特征库 | 3 | 检查重复、冲突和是否已长期沉淀 |
| 日常观察 | 1 | 核对近期身体和日常状态 |

合计 4 张表、9 行，相关上下文约 6.5K 字符。

同一次整理测试：

- 聊天范围 21 条；
- 最终整理 Prompt 约 26K 字符；
- 包含只读相关表结构；
- 包含模型标签协议；
- 标签成功写入现有 `row.meta.tagBundle`。

当前备份中已有 rows 表的历史行已经全部带有标签；本版解决的是后续新增与更新时，模型必须稳定生成并维护这些标签。

## 7. 代码收敛结果

- `memory_table.js`：R0 约 4,547 行，本版约 4,476 行；
- 新增三个职责独立的服务模块；
- `facade.health()` 纳入新模块健康检查；
- 范围预览显示相关表、相关行和字符数；
- 审核页面显示本次跨表上下文摘要；
- R0 的扁平 UI 和整理抽屉保持不变，没有增加外层框。

## 8. 兼容边界

本版没有修改：

- IndexedDB schema；
- `.ee` 备份格式；
- 模板和字段结构；
- 现有 `tagBundle` 格式；
- 角色和聊天消息格式；
- 记忆任务、审核和回滚的基本协议；
- 普通聊天 Prompt 的档案注入逻辑；
- R0 扁平表格布局。

- 静态架构检查通过；
- 35/35 组 Node 回归测试通过；
- 真实 `.ee` 八表跨表装配运行通过；
- 标签 XML 解析和现有 `tagBundle` 写入通过；

- V2.10-R0—R6 与 V2.1—V2.9-R11 历史回归通过。

## 10. 明确边界

本环境没有使用真实外部模型密钥调用线上模型。因此已经验证的是：

- 整理 Prompt 的完整协议；
- 真实数据的相关表选择；
- XML 标签解析；
- 审核和写入逻辑；
- 历史回归。

不同模型是否严格遵守标签 XML，仍应在部署后通过一次真实整理操作核验。解析器会忽略缺失或不合法标签，而不会破坏原记忆行。
