# 章鱼机 V2.14-R8.1：直接写入分流闭环修复说明

## 目标

R8.1 是针对真实备份审计发现的问题制作的修正版，不提前扩展 R9 生命周期功能。

修复对象不是“把审核全部关闭”，而是让以下各层真正使用同一套最终决策：

```text
每字段证据与置信度
→ 字段写入策略
→ 当前角色表格覆盖
→ 直接写入 / 会话运行态 / 候选审核
```

本版不删除字段、不重排现有表格，也不自动合并用户已有正式记忆。

## 一、Sidecar 改为逐字段证据

旧协议给整个当前状态块共用一个 `source` 和 `confidence`。只要整个块被模型标记为推断，其中来自用户原话的字段也会被当成推断，最终错误进入审核。

R8.1 的状态协议改为：

```json
{
  "status": {
    "fields": {
      "user_当前场景": {
        "value": "工作中",
        "evidence": "user_explicit",
        "confidence": 100
      },
      "user_精神状态": {
        "value": "轻度焦虑",
        "evidence": "assistant_inferred",
        "confidence": 82
      }
    }
  }
}
```

每个字段独立判断：

- 用户明确表达且达到阈值：按字段和表格最终策略处理；
- 模型推断的当前状态：写入会话运行态；
- 仅人工字段：继续阻止自动写入；
- 旧版全局 `source/confidence`：继续兼容读取，但不再要求模型生成。

## 二、当前状态推断不再堆审核

当前状态中的精神状态、体力、精力和风险等推断字段属于短时判断，不适合作为长期事实反复堆入审核队列。

R8.1 改为：

```text
用户明确表达 → 允许进入正式短期状态
模型推断 → 只进入 memoryTables.runtimeState.fieldValues
```

运行态仍可供当前会话使用，但不会冒充用户已经确认的事实，也不会增加正式记忆历史。

## 三、旧当前状态审核批次自动收敛

旧版本可能为同一批当前状态字段连续生成多个 `sidecar_field_policy` 审核批次。

R8.1 在下一次 Sidecar 写入事务中执行一次兼容迁移：

1. 找出当前状态表的旧 Sidecar 审核批次；
2. 每个字段只保留时间最新的建议；
3. 删除旧重复批次；
4. 按新字段证据策略重新路由；
5. 推断内容进入会话运行态；
6. 整个操作通过写入网关执行，保存失败时恢复完整快照。

这不是删除正式档案。真实备份验证中，9 个旧批次收敛后，正式记忆数据哈希保持不变。

## 四、角色“直接写入”覆盖候选通道

旧 Sidecar 候选通道无条件先加入候选队列，没有读取 R5 已经计算出的当前角色最终策略，因此“近期经历”和“日常观察”即使被当前角色设为直接写入，也仍然要求人工处理。

R8.1 新规则：

```text
目标表最终策略为直接写入
+ 候选来自用户明确表达
+ 候选置信度达到每个字段阈值
+ 所有实际写入字段均允许直接写入
= 自动 Upsert 正式记忆
```

以下情况继续进入候选：

- 模型推断；
- 置信度不足；
- 目标表不是直接写入；
- 任一实际字段要求审核、仅人工或被阻止；
- 目标表失效。

系统自动生成的事件 ID、更新时间等技术字段不会因为 `aiEditable:false` 错误阻断整条记录。

## 五、日常观察按日期 Upsert

日常观察的业务语义是“同一天优先合并为一条”，但旧通用身份规则没有把日期定义为该表的强身份，因此同一天仍可能新增多行。

R8.1 为 `daily_observation` 增加专用身份：

```text
recordKey = 日常观察表 + 标准日期
```

同一天再次写入时更新原行，不新增第二行。日期无法规范化时，才回退到来源指纹和内容指纹等通用规则。

## 六、设置页面显示字段实际分流

过去表格只显示表级“直接写入”，容易让人误以为所有字段都无条件直接。

R8.1 在有效策略摘要中增加字段路由统计：

```text
直接 N
审核 N
候选 N
运行态 N
仅人工 N
阻止 N
```

因此可以同时看到：

- 当前角色对整张表的最终覆盖；
- 表内每个字段实际会走哪条路。

## 七、停止注入旧 runtime-only 正式值

历史数据中可能仍有 `char_*`、角色判断、回应策略等字段值存放在正式 KV 数据内。虽然字段已经设置为 `runtime_only`，旧上下文组装器仍可能回退读取这些正式值。

R8.1 统一修正：

- 上下文组装；
- 召回文本；
- 搜索文本；
- 更新比较；
- 页面显示值。

`runtime_only` 字段只读取运行态值；没有运行态值时为空，不再默认回退到旧正式值。旧值暂时保留在数据中，避免破坏性迁移，但不会继续作为用户正式记忆注入。

## 八、实际备份验证

使用用户提供的单人精简备份副本验证：

```text
模板：1
表格：8
字段：94
正式多行记忆：279
旧当前状态 Sidecar 审核批次：9
```

结果：

```text
旧当前状态审核批次：9 → 0
保留的最新字段建议：8
迁入运行态的当前状态字段：11
审核迁移前后正式记忆哈希：一致

近期经历：31 → 32（明确事实直接写入）
日常观察首次同日写入：8 → 8
日常观察第二次同日写入：8 → 8
同日重复新增：0
自动晋升候选：3
源备份文件被修改：否
```

验证脚本只操作内存副本，不改用户原始备份。

## 九、本版边界

R8.1 不做：

- 自动删除或重排现有 94 个字段；
- 自动拆分“待办”和“近期经历”表；
- 自动删除历史 `char_*` 正式值；
- 自动合并旧日常观察重复行；
- 生命周期归档与冲突合并；
- 来源变化链 UI。

字段结构的收敛建议单独记录在《V2.14-R8.1 字段结构审计与迁移建议》中，后续必须通过预览、备份和可回滚迁移实施。
