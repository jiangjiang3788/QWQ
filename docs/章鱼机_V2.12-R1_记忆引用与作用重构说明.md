# 章鱼机 V2.12-R1：记忆引用与作用重构说明

## 1. 本版目标

V2.12-R1 只处理“这一轮究竟引用了什么记忆、为什么引用、产生了什么作用”这一条产品主线。

原来的召回诊断、记忆使用反馈和管理入口虽然共享同一批数据，但分别以命中卡片、反馈卡片和管理入口呈现。用户需要在多处来回查找，且单条卡片先展示了大量技术字段，不能先回答以下核心问题：

1. 本轮引用了哪些记忆表；
2. 每张表引用了几条；
3. 为什么被引用；
4. 在回复中承担什么作用；
5. 引用结果是否有帮助。

本版将上述能力收敛为一个领域和一个入口：**记忆引用与作用**。

本版不修改悬浮球操作结构，不修改模板与 JSON 编辑器，也不改变记忆表 schema、备份格式或召回算法。

---

## 2. 用户界面

### 2.1 单一管理入口

管理页只保留“记忆引用与作用”。旧的 `retrieval` 和 `feedback` 路由继续兼容，但会统一映射至：

```text
manage / usage_audit
```

待处理队列中的待反馈项目也打开同一个页面，不再进入另一套反馈卡片页。

### 2.2 先按表看，再看记录

页面信息顺序固定为：

```text
轮次与本轮检索线索
        ↓
引用表总览
        ↓
引用记录明细
        ↓
折叠的技术信息和策略设置
```

“引用表总览”使用标准表格，列出：

| 字段 | 含义 |
|---|---|
| 来源表 | 记忆来自哪张表 |
| 引用 | 本轮实际注入多少条 |
| 为什么引用 | 归一化后的召回原因 |
| 用于什么 | 对本轮回复承担的作用 |
| 相关度 | 该表引用记录的平均分数 |
| 反馈状态 | 待反馈、有帮助或需调整的数量 |

“引用记录明细”进一步列出：

| 字段 | 含义 |
|---|---|
| 来源表 | 记录所属表和模板 |
| 引用记录 | 标题、正文摘要和稳定行 ID |
| 引用原因 | 标准原因与原始依据 |
| 本轮作用 | 当前状态、关系边界、历史背景等用途 |
| 相关度 | 单条记录评分 |
| 反馈 | 有帮助、无关、过时、不准确及更多处理 |

### 2.3 召回原因标准化

原有召回原因文本继续保留，但显示层先归纳为用户可以理解的类型：

```text
固定附带
语义相似
关键词匹配
标签/场景
历史反馈
时效/重要度
来源证据
作用类型
规则命中
```

这样用户先看到“为什么”，需要时仍可查看更细的原始原因。

### 2.4 作用类型标准化

现有 `effectMode` 没有被替换，而是映射成用户语言：

| 原作用 | 页面显示 |
|---|---|
| fact | 确认事实 |
| temporary_state | 当前状态 |
| soft_preference | 回应偏好 |
| hard_boundary | 关系边界 |
| reminder | 提醒事项 |
| historical_context | 历史背景 |
| candidate | 候选参考 |

### 2.5 反馈仍在引用记录旁完成

没有新建反馈数据库。每条记录继续使用现有反馈快照与事件系统，可直接标记：

- 有帮助；
- 无关；
- 过时；
- 不准确；
- 当前场景禁用；
- 不要主动提及；
- 暂停使用；
- 忘记；
- 重置反馈。

反馈完成后仍参与后续相关性、冷却、时效和场景策略。

### 2.6 手机端

桌面使用标准表格；窄屏下同一张语义表转为两列字段布局，不创建另一套移动数据模型。

桌面和手机保持相同字段名称、相同轮次、相同反馈动作。页面级宽度不溢出；较长信息在记录内部换行。

## 3. 架构收敛

### 3.1 新增独立领域模块

新增：

```text
js/features/memory/retrieval_audit.js
css/modules/memory_retrieval_audit.css
```

`retrieval_audit.js` 负责：

- 选择和读取可用反馈轮次；
- 在没有已完成轮次时读取最后一次召回诊断；
- 将记录标题和正文摘要标准化；
- 将召回原因归一；
- 将作用模式转换为用户语言；
- 按模板和表格汇总；
- 生成总览与明细视图模型；
- 渲染审计表格。

模块不负责：

- 执行召回；
- 修改召回评分算法；
- 直接保存反馈；
- 修改记忆表数据；
- 修改 Prompt。

反馈动作继续由现有反馈服务和主事件入口处理，避免复制业务规则。

### 3.2 工作区路由收敛

`workspace.js` 增加统一视图 `usage_audit`，旧路由通过 `canonicalView()` 转换：

```text
retrieval ─┐
           ├─> usage_audit
feedback ──┘
```

这使旧链接和旧调用继续可用，同时生产界面只维护一个入口。

### 3.3 主控制器边界

`memory_table.js` 只负责：

- 将统一页面挂载到技术视图区；
- 接收轮次切换事件；
- 将现有反馈按钮事件交给反馈服务；
- 兼容旧打开方式。

总览分组、原因分类和 HTML 结构没有继续写入主控制器。

当前 `memory_table.js` 为 4,308 行，低于本版 4,310 行门禁。

### 3.4 单一事实来源

本版直接读取现有：

```text
memoryTables.feedback.rounds
memoryTables.lastRetrievalDiagnostic
item.reasons
item.effectMode
item.directive
item.feedback
```

没有创建：

```text
retrievalAuditHistory
usageAuditItems
behaviorRecords
```

因此不存在“召回页一份、反馈页一份、行为页再复制一份”的数据分叉。
