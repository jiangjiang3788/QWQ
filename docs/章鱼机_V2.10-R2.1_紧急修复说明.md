# 章鱼机 V2.10-R2.1 紧急修复说明

## 1. 修复目标

本版本只处理 V2.10-R2 实际使用中暴露的紧急问题，不增加新的产品范围：

1. 操作反馈和请求内容显示被截断；
2. AI 操作详情层层套框，最深内容被困在小滚动框；
3. 聊天生成与结构化记忆整理缺少每条消息的准确时间；
4. 结构化记忆待处理页点击“接受”后缺少明确反应；
5. “最终组合 Prompt”和“分条来源”容易被误解为模型请求发送了两次。

## 2. 反馈与请求内容

### 修改

- 单条结果字符串的操作记录上限由约 1.2 万字符提高到 8 万字符；
- 最终原始请求预览上限提高到 12 万字符；
- Prompt 来源单段上限提高到 6 万字符；
- 操作详情中的结果、来源和原始请求不再使用 160～280px 的内部小滚动框；
- 仍保留安全上限，避免操作中心重新成为大数据膨胀源。

### 用户表现

长反馈会在操作详情页面自然向下展开。只有超过安全上限时才会显示明确的截断说明。

## 3. 操作详情 UI 扁平化

原结构接近：

```text
操作详情
└─ 模型请求卡片
   └─ Prompt 来源卡片
      └─ 来源条目折叠框
         └─ 小型滚动正文框
```

修复后：

```text
操作详情
├─ 请求摘要
├─ 来源分类
│  └─ 扁平来源条目
├─ 最终合并结果（单独核对入口）
├─ 最终原始请求
└─ 结果反馈
```

同时增加详情全屏按钮。手机端全屏详情占据接近整个视口，正文只由外层页面滚动。

## 4. 消息时间

### 聊天生成

无论 OpenAI-compatible 还是 Gemini 请求路径，每一条历史消息都会携带：

```text
[消息时间：2026-07-21 14:21:03 UTC+02:00]
```

时间包含：

- 年、月、日；
- 时、分、秒；
- 当前时区偏移；
- 缺失时间时明确写为“时间未记录”。

原有“相隔超过 30 分钟”的时间段提示仍然保留。

### 结构化记忆整理

发送给记忆整理模型的历史文本调整为：

```text
[2026-07-21 14:21:03 UTC+02:00] 用户：消息正文
[2026-07-21 14:21:16 UTC+02:00] 角色：消息正文
```

审核页面的本次总结范围预览上限从 5,000 字符提高到 30,000 字符；完成后的审核历史预览提高到 12,000 字符。

## 5. 结构化记忆“接受”交互

“接受”现在被明确拆成两步：

1. 点击“选中接受”，该提案进入已选状态；
2. 点击页面底部“保存已接受项（数量）”，才正式写入档案。

修复内容：

- 按钮文字由“接受”改为“选中接受”；
- 选中后显示“已选中接受”；
- 没有选中项目时，保存按钮保持禁用；
- 选中后 Toast 提示继续点击“保存已接受项”；
- 保存过程中显示“正在保存…”；
- 保存成功显示实际应用数量；
- 保存失败恢复按钮，并显示具体错误；
- 结构化记忆页面事件只绑定一次，避免重复进入页面后同一点击被多个监听器处理。

这不是把“接受”改成不生效，而是把“选择决策”和“正式提交”明确区分，避免误操作。

## 6. 是否发送了两次聊天请求

### 结论

主聊天回复路径没有因为“组合视图”和“分条视图”发送两次。

同一次请求在 UI 中有两种观察方式：

- **分条来源**：解释角色档案、记忆、世界书和聊天历史分别来自哪里；
- **最终合并结果**：核对这些内容组装后的最终 Prompt。

它们读取同一条请求记录，不会分别调用模型。

本版本进一步增加：

- 标题“模型请求（实际网络调用）”；
- 明确显示本操作共有几次真实网络调用；
- 每条请求显示“第 1 次、第 2 次……”；
- 最终合并结果标记“核对视图 · 不是第二次发送”；
- 主聊天回复保留请求去重键和 1.2 秒去重窗口。

需要区分：聊天回复完成后，自动日记、结构化记忆、向量记忆或小剧场可能各自发起独立的后台模型请求。这些是 R2 已展示的子操作，不是聊天正文重复发送。

## 7. 数据兼容性

本次修复没有改变：

- IndexedDB 表结构；
- 角色数据格式；
- 聊天消息数据格式；
- 结构化记忆数据格式；
- 备份格式；
- Prompt Trace R1 协议版本。

可以直接覆盖部署 V2.10-R2。升级后建议清除站点缓存和 Service Worker，再刷新页面。
