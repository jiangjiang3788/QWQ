# 章鱼机 V2.10-R2 后台操作回执实施说明

## 1. 本版目标

V2.10-R2 解决的是“主回复已经完成，但用户不知道系统随后又做了什么”的问题。

上一版已经可以查看一次模型请求的真实来源；本版继续把聊天回复之后触发的自动工作挂到同一次主操作下，包括：

- 回复内结构化档案 Sidecar 应用；
- 自动日记总结；
- 自动结构化档案更新；
- 自动向量记忆总结；
- 角色主动创作小剧场。

用户现在能够在悬浮球中看到每项后台工作的执行结果，以及未执行时的明确原因。

## 2. 用户侧变化

打开某次“生成角色回复”的操作详情后，新增“后台工作”区域。每个子任务会显示：

- 功能名称；
- 当前状态；
- 成功结果；
- 跳过原因；
- 失败信息；
- 是否仍在运行。

典型回执示例：

```text
后台工作 4 项

应用回复内档案更新    已完成  已应用 2 项档案更新
检查自动日记总结      已跳过  未达到日记总结间隔
检查向量记忆总结      已完成  已新增 1 条向量记忆
角色创作小剧场        已跳过  本次概率未命中
```

点击子任务可以进入它自己的操作详情，查看阶段、模型请求、Prompt 来源和结果。

## 3. 状态设计

后台任务统一使用以下状态：

| 状态 | 用户含义 |
|---|---|
| 进行中 | 后台任务正在执行 |
| 等待中 | 已进入任务队列，尚未开始 |
| 已完成 | 实际执行并成功产生结果 |
| 已跳过 | 功能关闭、未达到间隔、没有待处理数据或概率未命中 |
| 失败 | 已尝试执行，但发生错误 |
| 已取消 | 用户主动取消 |
| 已中断 | 页面刷新前仍未结束 |

“已跳过”不是错误。它用于把过去静默的条件判断转换为用户可理解的回执。

## 4. 架构变化

### 4.1 显式父子操作

主回复与后台工作之间不再依赖“当前活跃操作”猜测关系，而是显式传递：

```js
parentOperationId
```

结构如下：

```text
生成角色回复
├─ 应用回复内档案更新
├─ 检查自动日记总结
├─ 检查结构化档案更新
├─ 检查向量记忆总结
└─ 角色主动创作小剧场
```

即使主回复先完成，后续子任务仍然能够正确挂到该主操作下。

### 4.2 Operation Runtime 升级

`js/modules/operation_runtime.js` 升级到 schemaVersion 3，新增：

- `startChild(parentId, type, options)`；
- `runChild(parentId, type, options, executor)`；
- `getChildren(parentId)`；
- `skip(id, reason, result)`；
- `recalculateParent(parentId)`；
- 根操作筛选 `list({ rootsOnly: true })`；
- 后台状态聚合 `background`。

父操作只保存子操作 ID 和状态计数，不复制子任务的 Prompt、结果或请求正文，避免再次制造递归冗余。

### 4.3 主操作后台聚合

父操作保存轻量统计：

```js
{
  total,
  running,
  queued,
  success,
  skipped,
  failed,
  cancelled,
  interrupted,
  settled,
  pending
}
```

每个子任务变化时自动重新计算，悬浮球可实时更新。

## 5. 首批业务接入

### 5.1 回复内档案 Sidecar

`chat_ai.js` 将模型回复中的档案更新处理登记为 `memory.sidecar` 子操作。

能够区分：

- 成功应用若干项变更；
- 没有 Sidecar 指令；
- 指令存在但没有实际变化；
- Sidecar 解析或应用失败。

Sidecar 失败不会吞掉已经生成的可见聊天回复，但会留下独立失败回执。

### 5.2 自动日记

`journal.js` 的自动检查登记为 `journal.auto`。

跳过原因包括：

- 功能未开启；
- 尚未达到消息间隔；
- 没有新的可总结消息；
- 已有相同任务执行中。

实际执行时会记录总结范围和最终结果。

### 5.3 自动结构化档案

`memory_table.js` 的自动检查登记为 `memory.table.auto`。

回执包含：

- 到期表数量；
- 实际更新数量；
- 等待审核数量；
- 无到期表时的跳过原因；
- 执行失败信息。

底层记忆请求仍通过既有 Memory API Adapter 和任务队列运行，没有建立平行记忆系统。

### 5.4 自动向量记忆

`vector_memory.js` 的自动检查登记为 `memory.vector.auto`。

模型请求显式绑定到该子操作，因此能够查看它自己的实际 Prompt、模型和结果，而不会误挂到主聊天请求。

### 5.5 自动小剧场

`theater.js` 的角色主动创作链路登记为 `theater.character`。

无论功能关闭、概率未命中、内容为空、请求失败或生成成功，都会产生明确回执。命中后复用同一个子操作，不会再额外生成重复操作记录。

## 6. 悬浮球 UI

`floating_ball.js` 新增：

- 主操作只出现在一级历史列表；
- 一级操作卡片显示后台任务聚合；
- 操作详情显示后台子任务列表；
- 子任务可点击进入详情；
- “已跳过”视觉状态；
- 后台任务变化时实时刷新；
- 子任务运行时悬浮球仍显示忙碌数量。

普通用户首先看到结果和原因；Prompt、原始请求和开发字段仍位于详情的后续层级。

## 7. 数据控制

本版继续遵守轻量记录原则：

- 父操作不复制子操作；
- 子操作只保存受控摘要和请求快照；
- 不复制聊天数组、角色对象或整个记忆数据库；
- 操作历史仍保存在 sessionStorage；
- 最多保留 100 次操作；
- 深度请求正文只保留最近少量记录；
- 清除历史时，若后台子任务仍在运行，会同时保留其父操作，避免形成孤立任务。

## 8. 兼容性

本版未改变：

- IndexedDB schema；
- 角色和聊天数据格式；
- 结构化记忆表格式；
- 世界书格式；
- 备份格式；
- 现有 Prompt 来源视图；
- 既有 AI Provider 请求格式。

R0、R1 的操作记录和来源视图继续兼容。

## 10. 下一版建议

V2.10-R3 建议进入“数据变化回执”：

- 消息新增或修改数量；
- 角色档案字段修改前后；
- 结构化记忆新增、更新、失效数量；
- 小剧场保存结果；
- 日记和向量记忆写入结果；
- 收藏、节点摘要等非模型写入。

这样操作中心将从“后台做了什么”继续进化到“最终改变了什么”。
