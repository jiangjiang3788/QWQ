# 章鱼机 V2.10-R0：AI 操作中心 MVP 实施说明

## 1. 本版目标

V2.10-R0 不重写现有聊天、记忆或小剧场系统，而是在现有代码上增加一层统一的“操作记录”。

用户打开悬浮球后，可以直接看到：

- 当前正在执行什么；
- 执行到了哪个阶段；
- 使用了哪个模型；
- 实际请求大约有多大；
- 请求是否成功、失败或被取消；
- 操作最终产生了什么结果。

本版同时改造 UI 和业务链路，不做只有工程人员能感知的底层重构。

## 2. 已完成范围

### 完整操作链路

以下功能已经具有从开始到结果的完整操作记录：

1. 私聊与群聊回复；
2. 后台回复；
3. 对话总结；
4. 手动生成文字或 HTML 小剧场；
5. 角色主动创作小剧场；
6. 手动更新结构化档案。

### 通用基础记录

其他已经通过 `OVOAIRequestRuntime` 发出的 AI 请求，会自动生成基础操作记录，包括：

- 功能来源；
- 任务名称；
- Provider 与模型；
- 请求状态；
- 请求字符量；
- 耗时；
- HTTP 或网络错误。

这些功能在 R0 中暂时还没有全部补充专属的产品阶段和数据变更说明，后续版本会逐项精细化。

## 3. 新增架构

新增文件：

```text
js/modules/operation_runtime.js
```

该模块提供两个公开入口：

```text
OVOOperationRegistry
OVOOperationRuntime
```

核心能力包括：

- `start()`：开始一次操作；
- `stage()`：更新用户可见阶段；
- `attachRequest()`：绑定真实模型请求；
- `updateRequest()`：同步请求状态；
- `complete()`：记录成功结果；
- `fail()`：记录错误；
- `cancel()`：取消操作及其运行中请求；
- `run()`：为现有异步函数提供轻量包装；
- `list()` / `get()`：供悬浮球读取历史和详情。

操作层不取代现有模块：

- `OVOAIRequestRuntime` 继续负责网络传输、超时、并发与取消；
- `OvoMemory` 与 MemoryTable 模块继续负责记忆业务；
- Proment 继续作为 Prompt 深度工作台；
- 小剧场继续使用现有生成和保存逻辑。

## 4. 悬浮球产品改造

悬浮球默认首页由工具面板改为“AI 操作中心”。

### 首页

展示：

- 当前操作或最近一次操作；
- 当前阶段；
- 运行状态；
- 使用的模型；
- 请求字符量；
- 运行耗时；
- 最近五次操作。

运行中的操作会让悬浮球显示数字，并提供取消按钮。

### 操作详情

展示：

- 操作状态与结果摘要；
- 阶段时间线；
- 模型请求列表；
- Provider、模型、来源、消息数量、耗时；
- 经过脱敏的实际请求内容；
- 结果摘要或错误信息；
- 复制操作报告。

### 原有工具

以下能力没有删除，而是移动到辅助入口：

- API 模型拉取与切换；
- Git 上传和恢复；
- Proment；
- 开发日志控制台。

## 5. 请求内容与隐私

请求快照在实际请求完成 Provider 格式处理后记录，因此展示的是接近真正发送给模型的请求体，而不是根据设置重新推测的内容。

安全控制：

- API Key、Token、Authorization、密码等字段自动替换为 `***`；
- URL 查询参数中的密钥由 AI 请求运行时脱敏；
- 单次请求正文最多保留约 18,000 字符；
- 超长内容明确标记为已截断；
- 不保存请求 Header；
- 不保存完整数据库对象。

## 6. 存储策略

操作记录保存在 `sessionStorage`，不会写入角色数据库或结构化记忆数据库。

当前策略：

- 最多保留 100 次操作；
- 最近 12 次保留请求正文预览；
- 更早记录只保留摘要；
- 页面刷新前仍在运行的记录标记为“已中断”；
- 用户可以清除已完成历史；
- 不会进入普通数据备份，因此不会制造新的长期数据膨胀。

## 7. 首批业务接入方式

### 聊天

`getAiReply()` 会创建聊天操作，并记录：

```text
准备聊天上下文
读取角色档案与长期记忆
发送模型请求
处理模型返回
完成并写入聊天
```

聊天请求显式携带 `operationId`，不会与同时运行的其他任务混淆。

### 小剧场

手动小剧场与角色主动创作分别登记为：

```text
theater.generate
theater.character
```

角色主动小剧场能够关联正在运行的聊天操作作为父操作。R0 的 UI 暂时仍以独立记录展示，后续会增加父子树视图。

### 结构化档案

手动更新按钮通过 `OVOOperationRuntime.run()` 包装原有 `updateSelectedMemoryTable()`，不改动记忆任务队列、审核和写入逻辑。

已通过：

- 所有 JavaScript 语法检查；
- 全量静态架构检查；
- 备份服务与旧编码兼容检查；

UI 烟雾测试验证：

- 悬浮球正常打开；
- 当前运行数量实时变化；
- 操作详情正常展示；
- 实际请求内容可以展开；

- 完成后悬浮球恢复空闲状态。

尚未执行真实外部模型 API 的端到端调用，因此不同供应商的线上响应兼容性仍需要部署后实际试用确认。

## 10. R0 的明确边界

本版还没有完成：

- 按 Prompt 来源拆分角色档案、结构化记忆、世界书和聊天历史；
- 展示记忆为何被选中或排除；
- 展示每次操作修改前后的数据差异；
- 将自动日记、向量记忆、自动结构化档案整理显示为聊天操作的子步骤；
- 父子操作树；
- 长期持久化的操作历史；
- 操作级重试和回滚。

这些不是遗漏，而是下一阶段的范围控制，避免第一版同时大改 Prompt、记忆和 UI，增加回归风险。

## 11. 下一版建议：V2.10-R1

R1 聚焦“发送内容为什么这样组成”：

1. 建立 Prompt 片段来源协议；
2. 把私聊实际请求拆成角色档案、用户档案、世界书、长期记忆、聊天历史、用户输入和输出要求；
3. 接入现有 Proment 快照和世界书诊断；
4. 让结构化档案、日记和向量记忆报告本次实际注入内容；
5. 在操作详情中提供“用户视图”和“原始请求视图”；
6. 保持现有 Prompt 生成结果不变，只增加来源标记和展示。
