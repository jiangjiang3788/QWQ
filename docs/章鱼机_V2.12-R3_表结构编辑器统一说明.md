# 章鱼机 V2.12-R3：表结构编辑器统一说明

## 1. 本版目标

V2.12-R3 只处理记忆模板结构配置的收敛，不增加新的记忆召回、治理或悬浮球功能。

本版针对的问题是：

- “模板编辑”“字段设计器”“JSON 模式”属于同一份结构配置，却拥有不同入口、不同弹窗和不同交互；
- 旧 JSON 模式像另一套记忆页面，用户难以判断修改的是数据还是模板；
- 字段、表格更新策略、注入策略和原始 JSON 分散，容易产生配置不一致；
- 主控制器同时承担两套模板编辑器的渲染、事件和保存逻辑，维护成本持续上升；
- 手机端在多个编辑弹窗之间切换，使用路径过长。

产品目标是：

> 只保留一个“表结构编辑器”，字段、表格设置和结构 JSON 均编辑同一个 schema 草稿，并通过同一个保存出口落库。

---

## 2. 用户入口收敛

模板管理列表现在只保留一个结构入口：

```text
编辑结构
```

当前记忆表工作区也提供：

```text
表结构
```

两处入口都会打开同一个编辑器。

已经从生产界面移除：

- 独立模板文本编辑弹窗；
- 独立可视化字段设计器弹窗；
- “可视化编辑 / JSON 编辑”两套入口；
- 记忆数据页面上可见的普通/JSON模式切换。

旧模式按钮只保留隐藏兼容节点，避免历史脚本或旧测试直接查找 ID 时崩溃；它们不再出现在用户界面，也不会切换到旧 JSON 页面。

---

## 3. 统一编辑器结构

编辑器只有一个窗口和一份草稿，内部使用三个平级视图：

```text
字段｜表格｜结构 JSON
```

### 3.1 字段视图

字段视图按当前表的字段组统一展示：

| 分组 | 字段名 | 类型 | 默认值 | 普通显示 | AI 编辑 | 摘要标签 | 操作 |
|---|---|---|---|---|---|---|---|

特点：

- 字段组是一级分区，不再作为字段下方的小字；
- 可直接修改分组名称，修改后立即重新归组；
- 字段的显示、AI 可编辑、默认值和摘要标签在同一行；
- 高级选项、数值范围、AI 提示和条件规则按需展开；
- 新增、移动和删除字段均操作当前草稿。

### 3.2 表格视图

表格视图统一管理：

- 表格名称；
- 键值表 / 多行表模式；
- 核心、短期、中期、长期和审核层级；
- 自动更新开关与触发方式；
- 注入模式；
- 字段数量与字段入口；
- 表格顺序和删除。

当前表的高级设置仍在同一视图下方，包括：

- 轮次与消息间隔；
- 单次读取上限；
- 是否允许删除；
- 主聊天 API / 总结 API；
- 相关 Top-K；
- 注入字符预算；
- 有效期；
- 提取规则；
- 更新附加规则。

这些设置继续写入原有 `updatePolicy` 和 `injectionPolicy`，没有创建第二套配置。

### 3.3 结构 JSON 视图

结构 JSON 不再默认展示一整块裸文本，而是转换成可编辑路径表：

| 路径 | 类型 | 值 |
|---|---|---|
| `tables.0.name` | text | 核心确认档案 |
| `tables.0.updatePolicy.enabled` | boolean | true |
| `tables.0.columns.0.group` | text | 核心确认 |

路径表覆盖：

- 模板名称和描述；
- 表格名称、模式和层级；
- 更新策略；
- 注入策略；
- 字段名称、分组、类型和默认值；
- 字段选项、AI 权限、摘要标签和条件规则。

原始 JSON 仍保留在折叠的高级区域，用于：

- 查看完整 schema；
- 批量粘贴或导入结构；
- 高级排错。

应用原始 JSON 后，会先经过同一个 schema 规范化过程，再回到统一草稿，不会保存为平行配置。

## 4. 单一事实来源

三个视图共享：

```text
uiState.templateDraft
```

不存在“字段编辑草稿”“模板 JSON 草稿”“设计器草稿”三份副本。

本版浏览器专项实际验证了双向同步：

1. 在字段视图将首个字段组改为“审计统一分组”；
2. 切换到结构 JSON；
3. `tables.0.columns.0.group` 立即显示相同值；
4. 在结构 JSON 将 `tables.0.name` 改为“核心确认档案 · 同步核验”；
5. 切换到表格视图；
6. 表格名称立即显示相同值。

三个视图只在最终点击“保存结构”时调用同一保存流程。

## 5. 代码架构

新增两个领域模块：

```text
js/features/memory/schema_model.js
js/features/memory/schema_editor.js
```

### `schema_model.js`

只负责结构业务：

- 准备与规范化模板草稿；
- 统计表格、字段和分组；
- 字段分组；
- schema 路径读取与写入；
- JSON 路径表模型；
- 新增、删除和移动表格/字段；
- 原始 JSON 规范化。

它不访问 DOM，也不负责保存。

### `schema_editor.js`

只负责：

- 三个视图的 HTML；
- 编辑控件与 schema 角色映射；
- 表格和字段操作转发；
- 路径表编辑转发。

它不直接修改数据库，也不复制模板保存逻辑。

### `memory_table.js`

主控制器只保留：

- 打开和关闭统一编辑器；
- 编辑器会话状态；
- 事件委托；
- 最终规范化与保存；
- 保存后刷新记忆工作区。

已删除旧模板文本编辑器和旧字段设计器的大量生产路径。

| 版本 | `memory_table.js` |
|---|---:|
| V2.12-R2 | 4,308 行 |
| V2.12-R3 | 3,872 行 |
| 本版减少 | **436 行** |

新增模块合计 393 行，且业务模型和 UI 渲染已分开，后续不需要把结构配置重新堆回主控制器。

## 6. 数据兼容性

本版没有修改：

- 记忆模板 schema 的已有字段；
- `memoryTableTemplates` 存储位置；
- 角色与模板绑定关系；
- 结构化记忆实际数据；
- 标签、生命周期、关系与审核协议；
- `.ee` 备份格式；
- 模板包和记忆包导入导出格式；
- 模型整理 Prompt 协议。

保存前仍调用原有 `Domain.normalizeTemplate()`，因此旧模板和旧备份会沿用已有迁移、默认值和 ID 生成规则。

本版不自动修改用户模板。浏览器同步测试只操作备份模板的内存副本，没有写回原 `.ee` 文件。

## 9. 本版明确边界

本版解决的是结构编辑入口和 schema 事实来源，不是重新设计记忆模板内容。

暂未加入：

- 字段权限矩阵的全新协议；
- 拖拽式列布局；
- 模板版本分支与多人协作；
- schema 差异预览；
- 保存前的真实数据迁移模拟。

这些能力未来应进入独立的 schema 迁移与版本治理领域，不应重新塞进当前编辑器主视图。

---

## 10. 下一阶段建议

V2.12-R0—R3 已完成用户可见入口收敛。下一阶段应从“继续增加页面能力”切换为工程边界收口：

- 将模板持久化、导入导出和迁移验证从主控制器继续拆出；
- 建立 schema 变更影响分析；
- 为大模板增加局部渲染与变更摘要；
- 建立页面层级、控制器体积和重复入口门禁。
