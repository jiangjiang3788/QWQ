# 章鱼机 V2.10-R3.3 任务稳定与记忆核验修复说明

## 1. 本版目标

本版针对四个实际使用问题进行稳定性修复，不继续扩展新的 AI 功能：

1. 离开聊天或记忆页面后，运行中的任务容易消失或被错误取消；
2. 悬浮球操作中心仍像浮层，打开后希望直接全屏并覆盖悬浮球；
3. 记忆使用反馈积压过多，较旧轮次已经失去反馈价值；
4. 需要确认结构化档案、向量记忆与聊天历史是否真的进入同一次模型请求。

## 2. 聊天任务与页面解耦

### 2.1 原因

旧代码使用全局变量：

- `currentChatId`
- `currentChatType`
- `currentReplyAbortController`
- `isGenerating`
- `saveCurrentChat()`

这些变量描述的是“用户当前看到的页面”，却同时被用于描述“异步任务真正属于哪个聊天”。

用户离开聊天页、返回首页或打开另一个聊天后，页面状态会改变。网络请求可能仍然执行，但后续保存与界面更新会引用新的全局状态，从而出现：

- 回复已经生成，但没有写回原聊天；
- 打开另一个聊天后，原任务被界面误判为空闲；
- AbortController 被新会话覆盖；
- 返回原聊天后看不到输入中状态；
- 数据已保存但 UI 没有即时显示。

### 2.2 修复

新增按聊天隔离的任务注册表：

```text
OVOChatReplyTasks
└─ private:characterId
   ├─ controller
   ├─ operationId
   ├─ startedAt
   └─ displayName
```

每次聊天任务持有自己的局部 AbortController，不再使用一个全局控制器承载全部聊天。

所有异步写入改为按明确对象 ID 保存：

```text
saveCharacter(chat.id)
saveGroup(chat.id)
```

不再依赖 `saveCurrentChat()` 判断当前页面。

消息气泡只在目标聊天仍处于打开状态时增量显示；无论界面是否打开，消息都会写回正确聊天。重新进入该聊天后，由数据库历史正常渲染。

### 2.3 页面切换行为

- 切换到首页、设置、记忆页或另一个聊天：任务继续运行；
- 返回原聊天：输入中状态和暂停按钮自动恢复；
- 打开另一个聊天：不会覆盖原任务的 AbortController；
- 同一聊天已有回复任务时：阻止重复启动；
- 不同聊天可以拥有各自独立的任务。

### 2.4 关闭或刷新浏览器

普通网页无法在浏览器真正关闭后继续保留尚未返回的网络连接。因此本版增加离开警告：

> AI 或记忆任务仍在运行。留在当前页面可继续完成；关闭或刷新会中断网络请求。

内部页面切换不会出现该警告，也不会取消任务。

## 3. 结构化记忆任务全局恢复

### 3.1 原问题

任务队列的中断恢复已经存在，但自动续跑主要发生在结构化记忆页面渲染时。用户刷新页面后如果不再进入记忆页，排队任务可能一直保持等待状态。

### 3.2 修复

应用完成数据加载和模块初始化后，会全局扫描所有角色：

- `autoResume = true`
- 队列未暂停
- 存在可运行的 queued 任务

符合条件的任务会自动恢复，不要求用户先进入记忆页面。

任务依旧按角色顺序执行，并继续使用原有的：

- 持久化队列；
- 失败重试；
- 中断恢复；
- 单轮 API 上限；
- 任务去重；
- 审核等待状态。

父任务和任务结果不会复制完整聊天或运行态。

## 4. 悬浮球全屏操作中心

打开悬浮球后现在直接进入覆盖式全屏页面：

- 覆盖整个视口；
- 打开期间隐藏原悬浮球；
- 页面主体禁止背景滚动；
- 顶部固定显示返回按钮；
- 返回按钮关闭操作中心并回到原页面；
- 操作详情、Proment、API/Git 和开发日志都在同一全屏容器中切换；
- 子页面返回时先回操作中心主页，不直接关闭整个页面。

安全区使用 `env(safe-area-inset-*)`，兼容手机刘海和底部手势区。

## 5. 记忆使用反馈过期治理

### 5.1 新策略

记忆使用反馈是针对“某一轮聊天中实际使用了哪些档案”的即时反馈，不应永久积压。

默认规则改为：

- 只保留最近 **3 轮**为可反馈状态；
- 更早的待反馈轮次自动标记为过期；
- 待反馈最长有效期默认 **7 天**；
- 请求处于 `prepared` 但超过 **2 小时**仍未完成，视为中断快照并自动失效；
- 过期项目不再计入待处理数量；
- 页面默认隐藏过期轮次；
- 提供“清理过期反馈”按钮。

用户可调整：

- 待反馈有效天数；
- 最多待反馈轮次；
- 页面显示轮次；
- 总快照保留数量。

### 5.2 不会删除档案

过期的是“请评价本轮记忆使用效果”的任务，不是档案记忆本身。

自动过期不会改变：

- 结构化记忆行；
- 角色档案；
- 记忆权重；
- 召回策略；
- 已经提交的有用、无关、过时等反馈。

### 5.3 实际备份结果

当前清理版备份的主角色共有：

- 11 轮已完成但待反馈；
- 5 轮未完成快照；
- 共 16 轮积压。

应用 R3.3 默认策略后：

- 保留最近 3 轮可反馈；
- 8 轮因已有更新轮次而过期；
- 5 轮因未完成超过 2 小时而过期；
- 总计自动收起 13 轮。

## 6. 聊天同请求记忆核验

### 6.1 核验位置

R3.2 已经在 system prompt 层增加结构化档案防丢失。本版进一步把检查移动到最靠近网络调用的位置：

```text
组装聊天与 Prompt
→ Provider 格式归一化
→ 合并联网搜索或自定义 payload
→ 最终 requestBody 记忆核验
→ 记录 memoryPayloadAudit
→ OVOAIRequestRuntime.request（唯一主聊天网络调用）
```

因此核验对象不是设计预览，也不是中间 Prompt，而是实际发送前的最终 Provider 请求体。

### 6.2 Provider 支持

OpenAI-compatible：

```text
requestBody.messages[role=system].content
```

Gemini：

```text
requestBody.system_instruction.parts[].text
```

如果结构化档案已启用、绑定有效模板，但最终请求体中没有 `<structured_archive_memory>`，系统会补入一次；补入后仍不存在则终止请求并报告错误，避免静默发送无档案聊天。

### 6.3 同一次请求

悬浮球新增“本次聊天记忆核验”，展示：

- 结构化档案是否发送、字符数；
- 向量记忆是否发送、字符数；
- 回忆日记是否发送、字符数；
- 实时状态与待办是否发送、字符数；
- 是否触发最终防丢失补入。

这些内容与聊天历史属于同一个主请求。结构化档案注入本身不会额外调用模型。

### 6.4 实际备份核验结果

主角色实际配置：

- `memoryMode = vector`；
- 结构化档案已启用；
- 已绑定 1 个有效模板；
- 当前结构化档案上下文 2,880 字符；
- 向量条目 0；
- 收藏日记 0。

使用 R3.3 实际最终请求体函数验证：

| Provider | 结构化档案 | 字符数 | 标签数量 |
|---|---:|---:|---:|
| OpenAI-compatible | 已发送 | 2,880 | 1 |
| Gemini | 已发送 | 2,880 | 1 |

主聊天代码路径只有 1 次 `OVOAIRequestRuntime.request()`。选择向量补充但向量为空时，结构化档案仍在同一聊天请求中发送。

## 7. 数据兼容

本版没有修改：

- IndexedDB schema；
- 角色、聊天、世界书格式；
- 结构化档案、向量记忆、日记格式；
- 备份文件格式。

新增状态主要存在于运行时、sessionStorage 和已有记忆反馈对象的兼容字段中。

## 8. 部署后验证

建议依次验证：

1. 在聊天 A 发送消息，模型生成期间返回首页，再进入聊天 B；聊天 A 应继续完成并保存；
2. 返回聊天 A，消息应存在且任务状态正确；
3. 创建一个结构化记忆任务，离开记忆页面，任务应继续；刷新后排队任务应自动恢复；
4. 打开悬浮球，应直接全屏，原悬浮球被覆盖，顶部返回按钮可关闭；
5. 打开记忆使用反馈，应只保留最近 3 轮为待反馈；
6. 发送一条私聊，悬浮球“本次聊天记忆核验”应显示结构化档案已进入同一次请求；
7. 最终原始请求中搜索 `<structured_archive_memory>`，应只有一组标签。

## 9. 边界说明

本地自动测试和实际备份 payload 构造均已通过，但没有使用用户真实 API 进行外部端到端调用。浏览器真正关闭或刷新时，未完成的聊天网络响应无法跨页面生命周期恢复；本版通过离开警告防止误操作，结构化记忆持久化队列则可以在刷新后继续恢复。
